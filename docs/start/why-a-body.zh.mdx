title: 为什么需要硬件？
description: 为什么 PhysiClaw 非得是一台实体机器——在你动手搭建之前，先讲清楚"真手点真屏"这件事为什么不可或缺。
---

PhysiClaw 是一个能操作手机的智能体——用相机看屏幕，用触控笔点划。为什么是手机？又为什么要用这么"笨"的物理方式？下面分三点说明。

## 大多数常用 App 没有 API

最直接的办法，当然是让智能体调用 App 的开放接口。但真正有价值的服务都是封闭的。淘宝、京东、美团、滴滴、微信——它们要的是用户停留在自己的界面里：多花时间、多消费、多看广告，并和用户保持直接联系。开放 API 与此背道而驰——智能体代替用户完成一切，界面被跳过，广告无从展示，平台自己也沦为一个随时可被替换的后端供应商，既丢了停留时长和广告收入，也丢了与用户的直接触点。所以它们没有动机向第三方智能体开放，反而有充分理由把外部访问挡在门外；你最想用的那些 App，很多根本没有公开接口。

在可预见的未来，没有谁会把那些主宰你日常生活的服务，免费、开放地交到你的智能体手上。这条路一开始就走不通。

## 软件模拟输入也走不通

接口走不通，自然会想：那能不能用软件模拟人的操作——模拟鼠标点击、键盘输入，或者手机手势？

在电脑端，程序可以模拟鼠标和键盘，但网页的反爬虫、反机器人检测愈发严格，自动化访问很容易被识别、被封，难以稳定使用服务。

在手机端，安卓的 ADB 可以注入点击和手势，但这类调用同样能被检测到——它们带着软件操作的痕迹，更容易触发风控，轻则异常拦截，重则直接封号。

说到底，凡是从软件层发出的指令，平台都有办法识别。只要被认出"不是真人在操作"，就会被限制或封禁。

## 物理方式绕过访问限制

接口和软件模拟都走不通，就只剩最朴素的一条路：像人一样，用手去点屏幕。

这正是 PhysiClaw 的做法——把手机屏幕直接当成 API：用相机看屏幕，用触控笔做手势，直接和 App 交互。不接 ADB，不装任何东西，零侵入。在手机看来，这和真人的手指操作没有区别，因此无从检测，也几乎可以稳定访问任何 App。

代价是慢。点一下、滑一下都要好几秒，远不及软件调用的几毫秒。这是一个有意的权衡：用速度换通用性和稳定性。对一个替你处理日常事务的智能体来说，慢一点不要紧，能可靠地把事情办成才重要。
