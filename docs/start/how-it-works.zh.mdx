OpenClaw 已经让你熟悉了「大脑」那一半：助理读屏幕、做决策、然后行动。PhysiClaw 沿用同一个思路，只是动作变成了对一台*真手机*的*物理* tap。本页把一次完整的 tap 从头到尾走一遍，让你看清一帧相机画面是怎么变成机械臂运动的 —— 以及它为什么被设计成一个闭环、可验证的循环，而不是盲目的遥操作。

PhysiClaw 做的每个动作，都是同一个四阶段循环转一圈。把循环固定下来，正是系统可靠的原因：每个阶段只干一件事，而且都以**查看结果**收尾。这里没有「驱动机械臂然后碰运气」式的开环操作 —— 在亲眼看到屏幕发生变化之前，系统从不信任任何一次移动。

```text
   ┌──────────────────────────────────────────────────────────┐
   │                                                          │
   ▼                                                          │
 LOOK ─────────────► DECIDE ─────────► MOVE & TOUCH ──────► CHECK
 overhead camera     agent picks a     arm drives stylus    look again:
 + on-device vision  box + a gesture   to the box, taps     did it change?
 boxes every         (tap / swipe /    then lifts away       │      │
 element             long-press)                          yes│    no│
                                                             │      │
                                              next action ◄──┘      │
                                              retry / re-aim ◄──────┘
```

整套系统就**一台相机加一支机械臂**。没有第二台相机，也没有深度传感器 —— 可靠性来自*每次移动后都重新看一眼*，而不是更花哨的硬件。

## 1. 看（Look）

PhysiClaw 用俯拍的 USB 相机给屏幕拍一张照，再把它喂给设备本地的视觉模型：用 **RapidOCR** 识文字，用 **OmniParser V2**（一个 YOLO11m ONNX 图标检测模型）识按钮、图标和图片。输出的不是原始像素，而是一份整洁的**清单**，其中每个元素都已经带好了标签和框：

```text
id  kind   label              bbox [left,top,right,bottom]   conf
12  icon   "Clock"            [0.41, 0.55, 0.49, 0.63]       0.97
13  icon   "Settings"         [0.51, 0.55, 0.59, 0.63]       0.96
14  text   "Wednesday 14"     [0.30, 0.08, 0.70, 0.13]       0.99
```

一个 **bbox**（「边界框」）就是框住某个元素的一个矩形，用四个从 `0` 到 `1` 的数字表示 —— 也就是屏幕宽和高的比例，顺序为 `[left, top, right, bottom]`。`[0.41, 0.55, 0.49, 0.63]` 的意思是「从横向 41%、纵向 55% 处开始，到横向 49%、纵向 63% 处结束」。用比例而不是像素，正是让一切都可移植的关键：同一份清单在任何尺寸的手机上都讲得通。这套 `0`–`1` 的比例约定是承重的 —— MOVE 阶段会把它直接换算成毫米。

完整的视觉流水线见[它如何看见](/zh/concepts/how-it-sees/)。

## 2. 决策（Decide）

助理 —— Claude，或者你指给 PhysiClaw 的任何模型（自带模型）—— 读这份清单，挑出**一个框**加**一个手势**。要打开时钟，它就挑元素 `12` 并调用 `tap`。它从不跟电机坐标或像素打交道；它的全部词汇就是*点名一个框，点名一个动作*。

真实的手势集很小，也很物理：`tap`、`double_tap`、`long_press`、`swipe`，外加 `home_screen`、`go_back`、`force_quit` 这样的导航动作。一共 12 个工具 —— 完整列表见 [MCP 工具参考](/zh/reference/mcp-tools/)。

:::note[两种驱动方式]
做决策的助理可以是一个连到 PhysiClaw 的**外部 MCP 客户端**（Claude Desktop、OpenClaw 等），也可以是 PhysiClaw 的**内置助理运行时**。无论哪种，look→decide→move→check 这个循环都完全一样；驱动方式详见[运行模式](/zh/run/operating-modes/)。
:::

## 3. 移动并触碰（Move & touch）

这一半在软件里没有对应物，所以这里给出诚实的机械图景。

编排器取选中 bbox 的**中心**（仍是 `0`–`1` 比例），用一套校准好的**仿射映射**把它换算成机械臂的真实毫米坐标 —— 这是一个先缩放再平移的直线公式，叫做 `pct_to_grbl`。这套映射正是[校准](/zh/set-up/calibrate/)要拟合出来的：在已知的网格上逐点触碰，学出屏幕比例与机械臂位置之间的对应关系。

机械臂本身有两组运动：

- **X/Y 是手工搭的 CoreXY 龙门** —— 一个从零开始造的框架，由两台 **NEMA 17** 步进电机分担负载，把触控笔在屏幕平面上移动。（CoreXY 的意思就是两台电机一起拉同步带，合成 X 和 Y。）这*不是*现成的消费级笔式绘图仪，而是你自己搭的龙门。
- **Z（触碰）是一个 12V 推拉式电磁铁** —— 一个行程约 10mm 的快速电磁铁，不是 Z 轴舵机，也不是绘图笔。电容触控笔笔尖拧在电磁铁铁芯上。一次 `tap` 是干脆利落的一落一抬；`long_press` 只是把笔尖多按一会儿（约 1.2s）。

电磁铁用的是**先冲击后保持**（hit-and-hold）的方案：先用大电流猛击一下、做出牢固接触，再降到较小的保持电流，把笔尖稳稳压住而不嗡嗡作响。具体的冲击/保持数值见[手势](/zh/reference/gestures/)。

有两个可靠性相关的行为值得单独点出：

1. **忙锁。** 每个手势都持有一把锁，所以动作绝不会重叠 —— 机械臂走完一个移动，下一个才能开始。
2. **自动归位。** 每次移动后机械臂都把触控笔停到屏幕外，好让下一张照片不被遮挡，LOOK 阶段看到的是一帧干净画面。

:::caution[触控笔必须接大地]
要让 tap 在电容屏上被识别到，笔尖必须接到一个验证过的**大地接地**上。这是真实存在、对可靠性至关重要的要求，不是可有可无的细节 —— 接线细节见[硬件搭建](/zh/hardware/assembly/)。
:::

## 4. 检查（Check）

PhysiClaw **再看一眼**，生成一份新的清单。助理把它跟自己的预期作比较。有三种结果：

- **按计划变化了** → 进行下一个动作。
- **什么都没变** → tap 没点中，或者框选错了；重新瞄准，再试一次。
- **冒出了意料之外的东西**（弹窗、广告、权限请求）→ 那就是新状态而已。助理读一读，再重新决策一遍 —— 没有脆弱的脚本会因此崩掉。

这套「观察结果，再重新决策」的设计，就是可靠性的全部秘密。PhysiClaw 应对意外的方式跟人一样：看一眼，再试一次。正因如此，一台相机加一支机械臂就够了 —— 验证来自重新看，而不是来自更多传感器。

## 两种看法：`peek` 与 `screenshot`

LOOK 其实有两挡，助理按需要来挑：

| | `peek` | `screenshot` |
| --- | --- | --- |
| 来源 | 俯拍**相机** | 手机**自己**的截屏 |
| 速度 | ~4s | ~12s |
| 清晰度 | 对大多数目标够用 | 像素级精准 |
| 副作用 | 无（不改变状态） | **会改变状态** —— 会触发手机的截屏手势，而 app 可能察觉得到 |

`peek` 是默认选项 —— 快，而且不留痕迹。只有当目标太小、相机分辨不清，或者反光让画面没法读时，助理才会升级到 `screenshot`。
