PhysiClaw 能触碰手指能触碰的一切——也就是说，凡是*你*在一台已解锁手机上能做的事，它都能做。
这份触达能力正是整个项目的意义所在，也正是你在买下任何一个零件之前，必须先想清楚的那件事。
这一页特意排在搭建章节之前：这里的最坏情况，决定的不只是你运行时的行为，更是你该选哪台手机、
哪些账户、哪个网络。

助理看着屏幕，敲击玻璃。在手机看来，那支触控笔就是一根没有边界的指尖：没有应用沙盒，
没有权限弹窗，也没有任何"你确定吗？"挡在助理和设备上已有的一切之间。只要一个手持你已解锁
手机的人做得到，PhysiClaw 就做得到。

:::danger[把核心风险说明白]
PhysiClaw 对一台已解锁的手机拥有**完全的物理控制权**。即便它一个密码都不知道，它仍然可以：

- 打开你的**密码管理器**，读出保存的登录凭据；
- 通过短信收到一条 **2FA 验证码**（应用为了让你登录而发来的一次性验证码），并用它来重置某个密码；
- 在任何**已登录**的应用里花钱；
- 读取你的**短信、邮件和照片**，或者冒用你的身份发消息。

这里没有可以兜底的软件边界，因为按照设计，整条链路里根本就没有软件。机械臂就是一根手指；
一根手指没有 API 可供锁定。
:::

## 这个风险你早已熟悉——只是现在它有了一根手指

你从 OpenClaw 一路走来，早就知道一个自主助理会被引向歧途：一个被投毒的网页、藏在它读到的
某段内容里的一次提示注入，或者干脆就是一次糟糕的模型决策。在普通助理上，这类失败的边界
就是你交给它的那点访问权限。

而在 PhysiClaw 上，**同样的**失败现在握着一根真实的手指，落在一台已解锁的手机上。一个被攻破的
助理会继承完整的已解锁手机访问权限——上面方框里的一切。这就是无 API 设计的代价：让它能
[在任意应用上工作](/zh/start/introduction/)、且零集成的那件事，正是拆掉了那些惯常围栏的那件事。

## 心智模型：一台已解锁的手机落在陌生人手里

助理可能用到的其他每一条访问路径，都带着一道围栏。API 发放的是受限范围的令牌。无障碍接口
可以被撤销。越狱至少会暴露自己。PhysiClaw 故意拆掉了所有这些围栏——所以那道围栏只能是**手机本身**。

理解一次 PhysiClaw 会话的正确方式：**你正把自己已解锁的手机交给一个陌生人，然后转身走开。**
未必是个心怀恶意的陌生人——但他会一字不差地照做指令，而且可能被骗。如果那台手机里装着你的银行
应用、你的主力号码、你的密码库，你绝不会把它交给陌生人。所以，别这么做。

## 用一台备用机——这是设备选择，不是某个设置

真正的缓解措施不是某个你事后去拨动的开关，而是一台你一开始就挑选好的设备。把 PhysiClaw 跑在一台
**专用的工具机**上——一台便宜的备用机，彻底抹除过——只给它任务真正需要的东西。这台机器上不该有
任何你会介意被陌生人看到或使用的东西。

下面这两个设计选择不是额外的杂活。它们恰恰证明了这个项目认真对待了"用备用机"这个决定——
只有在一台你早已决定当作可丢弃的手机上，它们才说得通。手机的准备步骤见
[准备手机](/zh/set-up/prepare-the-phone/)。

### 为什么密码是 `111111`

PhysiClaw 的 `unlock_phone` 工具把密码硬编码为 **`111111`**——这是刻意为之，不是占位符。
一个写在源码里、用完即弃的密码，意味着真正的密码永远不必由助理键入、不必存进配置、也不必通过
日志或 git 历史泄露出去。运行时你把工具机的密码设成 `111111`（或者干脆关闭自动锁定）——这只有
在一台你早已当作可丢弃的手机上才说得通。

:::caution[助理能键入的东西，它同样能读取]
同样的逻辑不止于密码。任何助理可能需要*键入*的东西——一个卡号、一次登录——它同样能*读取*并
外泄。除非某个具体任务确实需要，否则把秘密留在手机之外；任务完成后就移除它们。
:::

### 为什么服务器绑定 `0.0.0.0:8048`

PhysiClaw 服务器绑定到 `0.0.0.0:8048`，以便手机的桥接页面能通过你的局域网访问它——这是有意为之。
代价是，位于 `http://<host>:8048/mcp` 的 MCP 端点能被同一网络上的任何东西触及。`physiclaw doctor`
会把这一点作为**提示信息，而非错误**呈现出来：

```text
bind: 0.0.0.0 (LAN-reachable; intended for the phone bridge)
```

把整套设备跑在你信任的网络上，并且**不要把端口 `8048` 暴露到公共互联网**。

## 一项物理安全事项：给触控笔接地

上面的风险都关乎助理能*触达*什么。还有一项关乎*搭建*本身、很容易被忽略：电容式触控笔的笔尖
**必须接到大地**。

:::caution[把触控笔接到一处确认无误的大地——绝不要接墙插座，也绝不要接控制板 GND]
把触控笔笔尖连接到一处确知良好的大地，比如一台已接地 PC 的机箱。**绝不要**把它接到墙插座，也
**绝不要**接到控制板的 GND。没有正确接地，敲击根本不会被识别——所以这既是可靠性事项，也是安全事项。
:::

这一页只是把它标记出来。实际的接线步骤在[组装](/zh/hardware/assembly/)里，到那里再做。

## 安全是成熟的默认选择，而不是恐惧

以上这些都不是你回避这个项目的理由。一个拥有真实物理触达能力的系统*本就应该*配上一份清晰、
诚实的威胁模型——而把最坏情况说清楚，恰恰是为了让你把整套设备搭建成"即使最坏情况发生也乏善可陈"：
一台抹除过的备用机、几块钱的余额、几个就算丢了也无妨的账户。这样做下来，你既能享受到一个可以使用
任意应用的助理的全部威力，又把影响半径控制在自己刻意选定的范围内。
