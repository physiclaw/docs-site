PhysiClaw is **both an MCP server and a complete agent** — two front ends on one
body. The arm, the camera, the calibration bundle, and the twelve MCP tools are
identical whichever way you drive it. The only thing that differs is the *brain*:
who chooses the bboxes. Either an external MCP client does the thinking and
PhysiClaw is just the hands, or PhysiClaw runs its own built-in agent and needs
no external client at all.

You already used the first mode in [your first task](/en/run/first-task/), where
Claude Desktop did the deciding. Here's how that compares to letting PhysiClaw
think for itself, and when you'd pick each.

## Two brains, one body

<CardGrid>
  <Card title="Bring your own agent (MCP)" icon="external">
    An external client — Claude Desktop, OpenClaw, your own script — connects to
    `http://localhost:8048/mcp` and calls the tools. **That client is the
    brain.** PhysiClaw is pure hands: it exposes `peek`, `tap`, `swipe` and the
    rest, and does exactly what it's told. You drive the conversation.
  </Card>
  <Card title="Built-in agent runtime" icon="rocket">
    PhysiClaw ships its **own** agent engine. Start the server and a runtime loop
    spawns alongside it, ready to wake on a trigger, plan, call the same tools,
    and act — no external client in the loop. The brain lives inside PhysiClaw.
  </Card>
</CardGrid>

The crucial thing: **the tools are identical either way.** The exact `peek` →
`tap(bbox)` → `peek` loop from the trace you watched runs the same whether an
outside client or the in-tree engine is choosing the boxes. In fact, the FastMCP
instructions PhysiClaw hands an external client at connect time and the doctrine
its own engine loads come from the *same file* (`agent/context/PHYSICLAW.md`) —
so both brains reason about the phone the same way. Neither mode is smarter or
faster; it's the same loop with a different driver.

## How they compare

| | Bring your own agent | Built-in runtime |
| --- | --- | --- |
| Who decides | external MCP client | PhysiClaw's own engine |
| What you need | an MCP client (Claude Desktop, etc.) | `physiclaw server` + a configured model |
| Starts on | you sending a message | a **trigger** firing (cron or poll) |
| Models | whatever the client runs | 6 in-process providers + `claude-code`; you pick the active ref |
| Memory & skills | the client's, if any | built-in persistent memory + a skill system under `~/.physiclaw` |
| Best for | interactive, sit-and-steer tasks | unattended, scheduled, recurring tasks |

Neither is "the real" mode — they're two front ends on the same hardware. Use the
external client when you want to sit and steer; use the built-in runtime when you
want the phone to do something on its own at 8am.

## Mode A — bring your own agent

This is the MCP-server face of PhysiClaw, and it's exactly what you did on the
previous page. The server exposes its tools over a streamable-HTTP endpoint, and
any MCP client connects and drives:

```text
  Claude Desktop / OpenClaw / your script   ← the brain
            │  MCP  (http://localhost:8048/mcp)
            ▼
        PhysiClaw  ← just the tools (peek, tap, swipe, …)
            │
            ▼
          phone
```

There's nothing extra to configure beyond the client itself: point its config at
the endpoint and talk to it. The agent's quality, memory, and personality are all
the client's; PhysiClaw contributes only the eyes and the finger, plus the FastMCP
instructions it hands over at connect time. The conversation **starts when you
send a message** — you're in the loop, watching and steering. The worked example
is [your first task](/en/run/first-task/).

## Mode B — built-in agent runtime

PhysiClaw is also a *complete agent* on its own. When you run `physiclaw server`,
it spawns the **runtime** as a sibling subprocess. That loop polls a set of
**hooks** on a timer (default `1.0s`); when a hook fires a **trigger**, the engine
wakes, plans, and calls the same MCP tools to get the task done. After each run it
sleeps `react_cooldown_seconds` (`6.0`) before polling again. No external client is
connected; the brain is inside.

```text
        cron / poll trigger fires
                  │
                  ▼
   PhysiClaw runtime  ← the brain (its own engine, picks a provider)
                  │  calls the same tools
                  ▼
              phone
```

Crucially, the loop only polls **once `/api/status` reports `ready: true`** — i.e.
calibration is complete — so it never tries to tap a phone it can't see.

### Choosing the engine

Mode B is selected by the `[agent] model` reference, which has the shape
`provider/model`. There are two engine paths:

- **`claude-code/<model>`** shells out to Anthropic's `claude` CLI as a
  subprocess. It reuses Anthropic's key and catalog but is *not* one of the
  in-process providers.
- **Any other provider id** runs the **in-process engine**. There are six
  in-process providers:

  | Provider id | Vendor / API |
  | --- | --- |
  | `qwen` | DashScope |
  | `moonshot` | Kimi |
  | `openai` | OpenAI |
  | `anthropic` | Anthropic direct API (`/v1/messages`) |
  | `google` | Gemini (OpenAI-compatible) |
  | `deepseek` | DeepSeek (stub) |

:::note[`anthropic` and `claude-code` are not the same thing]
The `anthropic` provider talks to the direct API at `/v1/messages` from inside
PhysiClaw's own engine. The `claude-code` path instead launches the external
`claude` CLI as a subprocess. They use the same key but are different engines —
pick `claude-code/<model>` only if you want the CLI to drive.
:::

Models are **live-discovered** from each vendor's `/v1/models`; there's no curated
list baked in. Run `physiclaw models list <provider>` to see what's available.

### Arming Mode B — the model-gating gotcha

:::caution[The server runs but the agent won't wake]
There is **no default model.** A plain `physiclaw server` with no model still
runs — for hardware setup and for manual or external-client MCP calls — but the
built-in agent **will not wake.** This is the #1 "why is nothing happening"
confusion. To arm Mode B you must configure a model:

```bash
physiclaw models key <provider>          # store the API key
physiclaw models use <provider/model>    # set the active model ref
```

Or set the `PHYSICLAW_MODEL` environment variable. Most config writes print
*"Restart `physiclaw server` to apply."* — so restart the server after setting
the model.
:::

### What wakes the runtime

The runtime starts on a **trigger** firing — not on you sending a message. Two
kinds:

- **Poll** — the `phone_watch` hook GETs `/api/phone/watch` and the agent wakes
  when the phone changes (a camera-wake — e.g. a new notification appears).
- **Cron** — the `cron` hook reads `~/.physiclaw/jobs/jobs.md` and fires due jobs
  on schedule.

Full depth on both is in [Autonomous tasks](/en/agent/autonomous-tasks/).

### What the runtime adds

- **Multi-provider engine** — the same agent loop runs on any of the six
  in-process providers plus the `claude-code` path, and you pick the active ref.
  See [Models](/en/agent/models/).
- **Trigger-driven wake** — it runs unattended, on schedules or polls instead of
  on your messages. See [Autonomous tasks](/en/agent/autonomous-tasks/).
- **Memory & skills** — persistent memory across sessions and a skill system under
  `~/.physiclaw`. See [Memory and skills](/en/agent/memory-and-skills/).

### Runtime controls

A few flags worth knowing now (full list in the [CLI reference](/en/reference/cli/)):

- `physiclaw server --no-runtime` — keep PhysiClaw a pure MCP server (Mode A
  only); no engine subprocess spawns.
- The runtime also won't spawn if no model is configured (the gotcha above).
- `physiclaw server --warm-start` — resume the saved calibration bundle instead
  of re-calibrating.

## When to use which

- **Mode A** — hands-on, interactive, sit-and-steer tasks where you watch and
  direct each move. Start here; it's what you already know.
- **Mode B** — unattended, scheduled, and recurring tasks: the "do something on
  the phone at 8am" case, or "react when a notification shows up."

Same hardware, same tools, same loop. Which brain drives is your call.

## Closing the OpenClaw loop

The [introduction](/en/start/introduction/) framed PhysiClaw as OpenClaw's idea —
let the agent use the same interface a human uses — pushed onto real glass. These
two modes are where that lands: PhysiClaw is **both an MCP server** (bring your
own brain, like pointing OpenClaw at it) **and a self-contained agent** (the brain
is in the box). Either way it's the same verified look → think → move → confirm
loop. Bring a brain, or use the one in the box.
