PhysiClaw **既是一个 MCP 服务器，也是一个完整的助理**——一具身体上的两副面孔。无论你用哪种方式驱动，机械臂、摄像头、校准包和那十二个 MCP 工具都是完全相同的。唯一不同的是*大脑*：由谁来挑选 bbox。要么由一个外部 MCP 客户端负责思考、PhysiClaw 只当一双手；要么 PhysiClaw 跑自己内置的助理，完全不需要外部客户端。

你在[第一个任务](/zh/run/first-task/)里已经用过第一种模式了，那次是 Claude Desktop 在做决定。下面就来看看它和让 PhysiClaw 自己思考相比是什么样，以及各自该在什么时候选用。

## 两个大脑，一具身体

    随时准备在某个触发器下被唤醒、规划、调用同一套工具、然后行动——回路里没有外部客户端。
    大脑就住在 PhysiClaw 里面。
  </Card>

关键之处在于：**两种方式下工具完全相同。** 你刚才看到的那段轨迹里的 `peek` →
`tap(bbox)` → `peek` 循环，无论是由外部客户端还是由内置引擎来挑选框，跑的都是同一套。事实上，PhysiClaw 在连接时交给外部客户端的 FastMCP 指令，和它自己引擎加载的行事准则，来自*同一个文件*（`agent/context/PHYSICLAW.md`）——所以两个大脑对手机的推理方式是一样的。没有哪种模式更聪明或更快；它就是同一个循环，换了个司机。

## 它们的对比

| | 接入你自己的助理 | 内置运行时 |
| --- | --- | --- |
| 谁来决策 | 外部 MCP 客户端 | PhysiClaw 自己的引擎 |
| 你需要 | 一个 MCP 客户端（Claude Desktop 等） | `physiclaw server` + 一个配置好的模型 |
| 启动时机 | 你发一条消息 | 一个**触发器**触发（cron 计划或轮询） |
| 模型 | 客户端跑什么就是什么 | 6 个进程内提供商 + `claude-code`；由你选当前用哪个 ref |
| 记忆与技能 | 客户端自带的（如果有） | 内置的持久记忆 + `~/.physiclaw` 下的技能系统 |
| 适合 | 上手即用、坐下来亲自操控的交互式任务 | 无人值守、定时、周期性的任务 |

没有哪个才是“真正的”模式——它们是同一套硬件的两个前端。想坐下来亲自操控时，用外部客户端；想让手机在早上 8 点自己做点什么时，用内置运行时。

## 模式 A——接入你自己的助理

这是 PhysiClaw 作为 MCP 服务器的那一面，也正是你在上一页做过的事。服务器通过一个 streamable-HTTP 端点暴露它的工具，任何 MCP 客户端连上来就能驱动：

```text
  Claude Desktop / OpenClaw / your script   ← the brain
            │  MCP  (http://localhost:8048/mcp)
            ▼
        PhysiClaw  ← just the tools (peek, tap, swipe, …)
            │
            ▼
          phone
```

除了客户端本身，没别的要配的：把它的配置指向这个端点，然后跟它说话。助理的质量、记忆和性格全都来自客户端；PhysiClaw 只贡献眼睛和手指，外加它在连接时交出的那份 FastMCP 指令。对话**在你发出消息时开始**——你在回路里，看着、操控着。完整的范例见[第一个任务](/zh/run/first-task/)。

## 模式 B——内置助理运行时

PhysiClaw 自身也是一个*完整的助理*。当你运行 `physiclaw server` 时，它会拉起 **运行时** 作为兄弟子进程。这个循环按定时器（默认 `1.0s`）轮询一组 **hook**；当某个 hook 触发一个 **触发器**，引擎就会唤醒、规划，并调用同一套 MCP 工具把任务做完。每跑完一次，它会睡 `react_cooldown_seconds`（`6.0`）再继续轮询。没有外部客户端连入；大脑在内部。

```text
        cron / poll trigger fires
                  │
                  ▼
   PhysiClaw runtime  ← the brain (its own engine, picks a provider)
                  │  calls the same tools
                  ▼
              phone
```

关键的是，这个循环只在 **`/api/status` 报告 `ready: true` 之后**才会轮询——也就是校准已完成——所以它绝不会去点一部它看不见的手机。

### 选择引擎

模式 B 由 `[agent] model` 这个引用来选定，它的形态是 `provider/model`。有两条引擎路径：

- **`claude-code/<model>`** 会把 Anthropic 的 `claude` CLI 作为子进程拉起来。它复用 Anthropic 的密钥和模型目录，但*不属于*进程内提供商。
- **任何其他 provider id** 走的是**进程内引擎**。一共有六个进程内提供商：

  | Provider id | 厂商 / API |
  | --- | --- |
  | `qwen` | DashScope |
  | `moonshot` | Kimi |
  | `openai` | OpenAI |
  | `anthropic` | Anthropic 直连 API（`/v1/messages`） |
  | `google` | Gemini（OpenAI 兼容） |
  | `deepseek` | DeepSeek（占位） |

:::note[`anthropic` 和 `claude-code` 不是一回事]
`anthropic` 提供商是在 PhysiClaw 自己引擎内部，直接对接 `/v1/messages` 的直连 API。而 `claude-code` 这条路径则是把外部的 `claude` CLI 作为子进程拉起来。两者用的是同一把密钥，但是不同的引擎——只有当你想让 CLI 来驱动时，才选 `claude-code/<model>`。
:::

模型是从各厂商的 `/v1/models` **实时发现**的；没有内置的精选清单。运行 `physiclaw models list <provider>` 即可查看有哪些可用。

### 武装模式 B——模型门控的坑

:::caution[服务器跑起来了，但助理不醒]
**没有默认模型。** 一个不带模型的、光秃秃的 `physiclaw server` 照样能跑——用于硬件设置，以及手动或外部客户端的 MCP 调用——但内置助理**不会唤醒**。这是排名第一的“为什么什么都没发生”的困惑。要武装模式 B，你必须配置一个模型：

```bash
physiclaw models key <provider>          # store the API key
physiclaw models use <provider/model>    # set the active model ref
```

或者设置 `PHYSICLAW_MODEL` 环境变量。大多数配置写入都会打印 *"Restart `physiclaw server` to apply."*——所以设好模型后，重启一下服务器。
:::

### 是什么唤醒运行时

运行时是在一个**触发器**触发时启动的——而不是你发消息时。有两种：

- **轮询** ——`phone_watch` hook 对 `/api/phone/watch` 发 GET，当手机发生变化时助理就唤醒（一次摄像头唤醒——例如冒出一条新通知）。
- **Cron** ——`cron` hook 读取 `~/.physiclaw/jobs/jobs.md`，按计划触发到期的任务。

两者的完整细节见[自主任务](/zh/agent/autonomous-tasks/)。

### 运行时补上的部分

- **多提供商引擎** ——同一个助理循环可以跑在六个进程内提供商中的任意一个上，外加 `claude-code` 这条路径，由你选当前用哪个 ref。见
  [模型](/zh/agent/models/)。
- **触发器驱动的唤醒** ——它无人值守地运行，靠计划或轮询唤醒，而不是靠你的消息。见
  [自主任务](/zh/agent/autonomous-tasks/)。
- **记忆与技能** ——跨会话的持久记忆，以及 `~/.physiclaw` 下的一个技能系统。见
  [记忆与技能](/zh/agent/memory-and-skills/)。

### 运行时控制项

有几个标志现在值得了解（完整清单见 [CLI 参考](/zh/reference/cli/)）：

- `physiclaw server --no-runtime` ——让 PhysiClaw 保持为一个纯粹的 MCP 服务器（仅模式 A）；不拉起引擎子进程。
- 如果没有配置任何模型，运行时同样不会拉起（就是上面那个坑）。
- `physiclaw server --warm-start` ——恢复已保存的校准包，而不是重新校准。

## 该用哪一个

- **模式 A** ——亲力亲为、交互式、坐下来操控的任务，你看着、指挥着每一步动作。从这里开始；这也是你已经熟悉的那套。
- **模式 B** ——无人值守、定时、周期性的任务：“早上 8 点在手机上做点什么”那一类，或者“有通知冒出来时做出反应”。

同一套硬件、同一套工具、同一个循环。由哪个大脑来驱动，是你的选择。

## 收束 OpenClaw 的闭环

[引言](/zh/start/introduction/)把 PhysiClaw 描述为 OpenClaw 这一理念——让助理使用人类所用的同一套界面——被推到真实玻璃屏上的产物。这两种模式正是这一理念的落脚点：PhysiClaw **既是一个 MCP 服务器**（你自带大脑，就像把 OpenClaw 指向它）**也是一个自给自足的助理**（大脑就在盒子里）。不管哪种方式，都是同一个经过验证的 看 → 想 → 动 → 确认 循环。自带一个大脑，或者用盒子里现成的那个。

接下来，去了解那个内置大脑里面有什么——[助理概览](/zh/agent/overview/)，然后[配好一个模型](/zh/agent/models/)并[安排定时任务](/zh/agent/autonomous-tasks/)。
