一次点击能落在该落的地方，是因为 PhysiClaw **一次性**学会了三个坐标系如何对齐：大脑的**屏幕分数**、
相机的**像素**，以及机械臂的**毫米**。标定就是那半分钟——机器在这段时间里测量这些关系，
把它们烘焙进两个小矩阵；此后每一次手势，无非是把一个边界框中心穿过它们做乘法。本页讲的是直觉，
外加一点点数学——写给看着[配置](/zh/set-up/calibrate/)跑完、想*理解*它而不是重新跑一遍的读者。

## 三个坐标系

一次点击碰到的一切，都活在三个空间之一里，而标定的全部工作就是在它们之间翻译：

```text
  screen 0–1            camera pixels          arm millimetres
 (what the brain        (what the lens         (what the gantry
  reasons in)            actually sees)         actually moves in)

 [0.55, 0.59]  ─────────── A (pct_to_grbl) ────────►  (gx, gy) mm
       ▲
       └──── B (pct_to_cam) ◄────  pixel  ────  (camera frame)
        listing comes back in 0–1, from camera pixels
```

- **屏幕 0–1** —— [边界框清单](/zh/concepts/how-it-sees/)使用的那套可移植分数。
  `(0,0)` 是手机屏幕的左上角，`(1,1)` 是右下角。大脑*只*讲这一种话。
- **相机像素** —— 某个东西出现在头顶画面里的位置。完全取决于你把相机装在哪儿，所以在测量之前
  毫无意义。
- **机械臂毫米** —— 龙门架实际驱动到的 GRBL 坐标。同样在测量之前毫无意义，因为它取决于手机
  摆在机械臂下的哪个位置。

PhysiClaw 学两张映射来打通它们。**映射 A**（屏幕 → 机械臂，叫 `pct_to_grbl`）让它能*动手*；
**映射 B**（屏幕 → 相机，`pct_to_cam`）让它能*看见*——把一个检测到的相机像素转回成清单里用的
`0–1` 屏幕分数。记住这个口诀：**A 用来动手，B 用来看见。**

## 一次点击走过的换算链

当大脑说 `tap [0.51, 0.55, 0.59, 0.63]` 时，真正的算术是这样的：

<Steps>

1. **边界框 → 中心。** 把四角取平均：`cx = (0.51 + 0.59) / 2 = 0.55`、
   `cy = (0.55 + 0.63) / 2 = 0.59`。你瞄的是框的正中。

2. **屏幕 → 毫米（映射 A）。** 编排器把这个 `(0.55, 0.59)` 穿过 `pct_to_grbl` 仿射，得到以毫米计的
   GRBL 坐标 `(gx, gy)`。

3. **移动并击下。** 龙门架快速移到 `(gx, gy)`；电磁铁触发它的
   [击下并保持](/zh/reference/gestures/)；笔尖落下。

</Steps>

那第二步就是 [12 个 MCP 工具](/zh/reference/mcp-tools/)和实体机器之间的桥梁。`tap`、`double_tap`、
`long_press` 和 `swipe` 全都汇入同一套换算：编排器拿到以 `0–1` 百分比表示的边界框中心，让它穿过
`pct_to_grbl`，然后才去调用 `arm.tap` / `arm.double_tap` / `arm.long_press` / `arm.swipe_to`。
大脑从来看不到毫米。

`peek` 那条路把同样的想法反着跑：头顶相机以*像素*看到屏幕，而**映射 B**把那些像素转回 `0–1`
屏幕分数，于是大脑读到的清单，和它将要点击的边界框处在同一个空间里。A 用来动手，B 用来看见。

## 仿射映射究竟是什么

映射 A 是一个**仿射变换**——能拉伸、旋转、错切、平移一个平面，同时让直线保持笔直的最简单映射。
它存成一个 2×3 矩阵，像这样作用：

```text
┌ gx ┐   ┌ a  b  c ┐   ┌ screen_x ┐
│    │ = │         │ · │ screen_y │
└ gy ┘   └ d  e  f ┘   └    1     │
```

就是六个数。`a b / d e` 这一块处理缩放、旋转和错切——屏幕 X 走一个单位，机械臂 X 要走多少，
以及手机相对龙门架有多歪。`c f` 那一列是偏移——屏幕 `(0,0)` 落在机械臂空间的哪里。六个数就够了，
因为一台平放在头顶相机下的手机，非常接近一个被正面平视的平面：没有透视畸变、也没有镜头畸变要建模，
只有一点倾斜加拉伸。这正是为什么寥寥几个测量点就能钉住整个屏幕。

## 学习映射 A —— 机械臂自己做的 18 次点击

机械臂看不见屏幕，所以它靠**点击并倾听**来求屏幕 → 机械臂映射：手机会报回每次触摸究竟落在哪里
（以屏幕 `0–1` 计），机械臂知道自己当时在毫米空间的哪个位置，每次点击就给出一对精确的
（毫米，屏幕分数）。配对够多就能定下那张仿射。

<Steps>

1. **探针三角（3 次点击）。** 在机械臂原点 `(0,0)` 点一下，再沿 X `+10 mm` 点一下，再沿 Y
   `+10 mm` 点一下。三个点是引导出一张*粗略*仿射的最少数量——刚好够大致预测屏幕其余部分在哪里。

2. **网格（15 次点击）。** 用那张粗略映射，机械臂走遍铺满屏幕的 3 列 × 5 行网格，逐格点击。
   每一次点击都给出又一对精确的（毫米，屏幕分数）。

3. **拟合 + 重定原点。** 全部 **18 个点**连同倾斜一起用最小二乘拟合出最终那张精确的映射 A。
   机械臂随后把屏幕中心重新声明为它的 `(0,0)`，于是整个坐标系锚定在手机上，而不是锚在龙门架
   碰巧回零的那个位置。

</Steps>

整个流程**大约 30 秒**——就是你看着机械臂在自己身上点出来的那张网格。

:::caution[笔尖必须接地，这一切才成立]
只有当触控笔笔尖连到一个验证过的大地时，手机才会把电容触摸认下来。如果[接地](/zh/hardware/assembly/)
缺失，点击就不被识别，也就没有（毫米，屏幕分数）配对报回来——标定根本无法开始。因为电磁铁是固定行程的，
没有触摸*深度*要找；一次没点中只意味着接触不稳，机器会重新击一下。
:::

这次拟合还顺带产出一个免费诊断：一个**倾斜比**，衡量手机相对机械臂行程有多歪。接近 `0` 说明手机
和龙门架是正的；数值偏高就是在警告你，继续之前先把它摆正。

## 学习映射 B —— 相机读取的 15 个点

映射 B（屏幕 → 相机）是反过来学的：手机*显示*一个已知图案，相机*看*。桥接页面点亮一张
**15 点网格**——位于已知屏幕分数上的亮红点——头顶相机检测出每一个点的像素位置，给出 15 对
（屏幕分数，相机像素）。这一步要先检测到**全部 15 个点**才会拟合。

这次拟合比映射 A 稍微更稳一些。PhysiClaw 同时拟合 `pct_to_cam` 映射和一张 **RANSAC 单应**——一种
鲁棒的拟合，会忽略偶尔检错的点，而不是任由它把整张映射带歪。平面直觉依然成立（头顶看到的手机屏幕
接近平面），但正是单应让相机映射能扛住一次杂散的反光或阴影。结果正是 `peek` 所需要的：任意相机像素
都能转回成清单里用的屏幕分数。

:::note[为什么截图那条路需要它自己的前置步骤]
在两个触摸步骤之前，向导的**视口偏移**（第 5 步，而且它*必须*排在最前面）会在一个已知位置闪出一个
橙色方块，并读回手机自己的截图。从中它推导出设备像素比（`dpr`）和状态栏／安全区偏移。这就把
「网页桥接页面以为某个点在哪里」和「它在手机原生截图里落在哪里」对上账，从而让 `screenshot` 那条路上
点击也照样准。这一步弄错了，映射 A 和 B 就会拿一个位置不对的参照去拟合——这正是它要卡在触摸步骤前面
的原因。
:::

## 停到屏幕外

每两个动作之间，机械臂都退到一个固定点位，就在手机**外**一点——屏幕分数大约 `(-0.1, -0.05)`，
也就是左上角再往左、往上一点。屏幕外的坐标在这里完全合法：把 `(0.55, 0.59)` 转成毫米的那张映射 A，
也照样把 `(-0.1, -0.05)` 转成毫米——仿射变换并不在乎这个点已经越过了屏幕边缘，矩阵乘法照常工作。
这个停靠点重要有两个原因：它让触控笔退出相机视野，好让下一次 `peek` 不被遮挡；它也是一个已知、
可复现的歇脚位置，重启之后机器可以重新锚定到它。

## 验证、漂移，以及何时该重新标定

最后那个测量步骤会**验证**结果：它点 5 个随机点，检查每一个是否落在目标的屏幕宽度 `0.015` 以内
——在一块 390 像素宽的屏幕上大约是 **5 个像素**——并在 **5 中 ≥ 2** 时通过。这也是向导里*唯一*
会把**标定包写到磁盘**的步骤，所以一次没通过验证的运行什么都不会留下。

两张映射都假设相机、机械臂和手机都精确地待在测点时所在的位置。任何挪动了机器某个部件的东西都会让
这个前提失效，点击便开始落偏：

<CardGrid>
  <Card title="机器被碰了" icon="warning">
    撞到了机械臂，或手机在支架里移了位，于是映射 A 现在错了——毫米不再对应同样的屏幕分数。
    点击会稳定地*偏*同一个量。
  </Card>
  <Card title="相机动了" icon="warning">
    碰了或重新对焦了头顶相机，于是映射 B 现在错了——像素映射到了错误的分数，所以早在机械臂
    动起来之前，大脑瞄的就是一个略微错位的目标。
  </Card>
</CardGrid>

破绽是一种**系统性**的偏差：点击朝同一个方向一致地偏开一段固定距离。那指向一张*过期*的变换——
修法是[重新标定](/zh/set-up/calibrate/)、重测那些点，让矩阵对上机器的新现实。朝不同方向**随机**乱偏
则是另一种问题：接触或接地不稳，而不是映射过期。把一切都装得稳稳当当，你就很少需要再碰标定。

:::tip[结果存在哪里]
拟合好的映射会作为标定包存到 `~/.physiclaw/calibration/bundle.json`（数据根目录保存
`calibration/{bundle.json,cache/}`）。这一个文件就是「一次性烘焙进去」真正的意思——而服务器只有在你用
`--warm-start` 启动时才会复用已保存的标定包；否则它会期待一次全新的标定。
:::
