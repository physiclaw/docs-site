title: 手势
description: 一次 tap、double-tap、long-press 或 swipe 如何变成电磁铁占空比脉冲和 GRBL G-code——规范的时序曲线。
---

你已经了解了"大脑"：一个 LLM 调用某个 [MCP 工具](/zh/reference/mcp-tools/)，比如 `tap` 或
`swipe`，并传入一个以 `0`–`1` 屏幕比例表示的 **bbox**。这一页讲的是"身体"——这些工具调用
实际上对装置做了什么。它是 **电磁铁时序的事实来源**：下文每一个常量都直接取自 `solenoid.py`
和 `arm.py`，凡是旧页面与此冲突的，以本页为准（见
[关于过时片段的提醒](#关于过时片段的提醒)）。

每个手势都是同样的双机协作：**X/Y 机械臂** 把触控笔驱动到 bbox 中心——通过经过校准的
`pct_to_grbl` 仿射变换，把百分比坐标转换成 GRBL 毫米——然后一个 **电磁铁（solenoid）**
让笔尖落到玻璃上再抬起。Z 轴 *不是* 伺服，也没有深度需要校准：笔尖停在固定的机械高度。
手势之间唯一变化的是 **时序**（以及击打电流与保持电流之分）。让一次 tap 成为 tap、让一次
long-press 成为长按的，是线圈保持通电的时长——而不是它按得多用力或多深。

## 一段话讲清电磁铁

触控笔的笔尖由一个 **12V 0.8A 推拉式电磁铁** 向下推动，它接到 GRBL/FluidNC 的
**主轴 PWM** 引脚（`gpio.32`），所以它用主轴 G-code 驱动：`M3 S<duty>` 以 `0`–`1000`
刻度上的某个 PWM 占空比给线圈通电，`M5` 释放它。线圈断电时一根回位弹簧抬起笔尖。
电磁铁若以满电流保持通电会烧坏线圈，所以 PhysiClaw 采用 **击打并保持（hit-and-hold）**
的曲线：先猛击把铁芯吸入，然后降到较低的保持电流以维持吸合状态。

## 规范时序

| 常量 | 取值 | 含义 |
| --- | --- | --- |
| `HIT_S` | `1000` | 击打占空比——把铁芯吸入（仅短暂吸入时使用） |
| `HOLD_S` | `750` | 保持占空比——唯一可安全 *持续* 维持的电平（long-press、swipe 滑动） |
| `SETTLE_MS` | `80ms` | 击打后、降到保持电平前的停留时间 |
| `RELEASE_MS` | `200ms` | `M5` 之后、任何 XY 移动之前的弹簧回弹停留时间 |
| `DOUBLE_TAP_GAP_MS` | `100ms` | 双击两次击打之间的短暂抬起 |
| `TAP_DURATION` | `0.08s` | 一次 tap 在玻璃上停留的时长 |
| `LONG_PRESS_DURATION` | `1.2s` | 一次 long-press 保持吸合的时长 |

:::caution[为什么 `HOLD_S` 低于 `HIT_S`]
击打占空比（`S1000`）只用于约 80ms 的吸入；持续维持它会烧坏线圈。`HOLD_S`（`S750`）
是唯一可安全持续维持的电平——而且它被有意设得 *偏高*：在更凉的 `S500` 下，笔尖会在击打→保持的
切换处松开，面板会把一次 long-press 读成单击。`S500` 对 *移动中* 的 swipe 没问题（滑动会重新
建立接触），但对静止保持却不行。如果 `HOLD_S` 不严格低于 `HIT_S`，驱动会 **拒绝构造**——
它的 `__init__` 会抛出 `ValueError`。
:::

每次 `M5` 之后，PhysiClaw 会停留 `RELEASE_MS`，让弹簧在下一次 XY 移动开始前把笔尖完全抬离——
否则笔尖会在屏幕上拖出一条线。

## Tap

一次瞬时按压：以 `HIT_S` 击打，保持 **tap 时长（80ms）**，然后释放。这次按压足够短，
所以不需要降低的保持电流——`S1000` 持续 80ms 就能给出清脆的接触，又不会烧坏线圈。

```text title="Tap — G-code"
G0 X.. Y..        ; arm moves to the bbox center
M3 S1000          ; strike (stylus down)
G4 P0.08          ; hold 80 ms
M5                ; release
G4 P0.2           ; spring-rebound dwell (tip clears the glass)
```

## Double-tap

两次击打之间只隔一次短暂的断触抬起（`DOUBLE_TAP_GAP_MS = 100ms`）——**不是** 完整的 200ms
弹簧抬离停留。整对动作作为 **一个手势** 运行（单个通电窗口），所以"按下到按下"的间隔远低于
iOS 约 300ms 的双击窗口。把两次独立的 tap（各带自己的 200ms 释放）串起来会把它推到约 280ms，
时序抖动随即会把这个手势拆成两次单击。

```text title="Double-tap — G-code"
M3 S1000          ; strike 1
G4 P0.08          ; hold 80 ms
M5                ; lift briefly…
G4 P0.1           ; …100 ms, just enough to break contact
M3 S1000          ; strike 2
G4 P0.08          ; hold 80 ms
M5                ; release
G4 P0.2           ; spring-rebound dwell
```

## Long-press

这里"击打并保持"的曲线名副其实：以 `S1000` 击打，停留 80ms，降到 `S750` 保持电流，
并在抬起前把笔尖保持吸合整个 **long-press 时长（1.2s）**。iOS/Android 在约 500ms 处识别长按；
1.2s 留出了充裕的余量。

```text title="Long-press — G-code"
G0 X.. Y..        ; arm moves to the bbox center
M3 S1000          ; strike
G4 P0.08          ; settle 80 ms
M3 S750           ; drop to hold current
G4 P1.2           ; hold 1.2 s
M5                ; release
G4 P0.2           ; spring-rebound dwell
```

## Swipe

swipe 会先按压，然后 *带载移动*。整个滑动过程中笔尖以 `HOLD_S`（`S750`）保持按下，机械臂从
起点到终点运行一段受控的 `G1` **直线** 移动（不是快速的 `G0`），然后抬起。

`direction` 指的是 **触控笔** 的运动方向，而不是页面的运动方向——向 *上* swipe 会让页面向 *下*
滚动。行程长度来自 `size` 参数（`_SWIPE_DISTANCES`），它映射为屏幕的一个比例；底层机械臂的默认值
（`SWIPE_DISTANCE`）是 15mm。

| `size` | 屏幕比例 | 用途 |
| --- | --- | --- |
| `s` | `0.10` | 小幅推动 |
| `m` | `0.30` | 大多数滚动 |
| `l` | `0.50` | 长滚动 |
| `xl` | `0.75` | 整页大小的滚动 |
| `xxl` | `0.90` | 全屏（控制中心 / 通知中心） |

`speed` 参数设定 `G1` 进给速率（`SWIPE_SPEEDS`）：

| `speed` | 进给 | 手感 |
| --- | --- | --- |
| `slow` | `F3000` | 谨慎滚动 / 拖动 |
| `medium` | `F6000` | 普通 swipe（约 100 mm/s） |
| `fast` | `F10000` | 快速甩动、翻页 |

```text title="Swipe — G-code"
G0 X.. Y..        ; arm moves to the start
M3 S1000          ; strike
G4 P0.08          ; settle 80 ms
M3 S750           ; drop to hold current (tip stays down for the slide)
G1 X.. Y.. F6000  ; drag to the endpoint at the chosen feed
M5                ; release
G4 P0.2           ; spring-rebound dwell
```

:::note[为什么由保持电流而非击打电流驱动滑动]
静止保持需要 `S750` 才能持续被识别，但一个 *移动中* 的笔尖在滑动时会不断重新建立接触——
`S500` 对 swipe 没问题，但对静止保持却不行。PhysiClaw 对两者都使用唯一可安全持续维持的电平
（`S750`），这样就只有一条保持路径需要推敲。
:::

## 小结

| 手势 | 按下 | 保持 | 移动 | 抬起 |
| --- | --- | --- | --- | --- |
| Tap | `S1000` | 80ms | — | `M5` + 200ms 停留 |
| Double-tap | `S1000` ×2 | 各 80ms，间隔 100ms | — | `M5` + 200ms 停留 |
| Long-press | `S1000` → `S750` | 1.2s | — | `M5` + 200ms 停留 |
| Swipe | `S1000` → `S750` | 滑动期间 | `G1`，`F3000/6000/10000` | `M5` + 200ms 停留 |

:::tip[安全网]
每个手势在返回前都会给线圈断电——即使发生异常或 Ctrl-C 也是如此（`try/finally` 加上一个
`_force_off` / `_energized` 上下文管理器）——所以 tap 中途发生故障也绝不会让线圈一直发热。
作为硬件兜底，FluidNC 固件（`off_on_alarm: true`）会在控制板进入报警状态时切断主轴 PWM。
:::

## 调试前提

要让上述一切生效，控制板必须处于 **主轴模式**（`$32=0`）。在激光模式下，固件只在 *移动时*
才输出主轴 PWM，所以一次静止的 `M3` 击打不会产生任何效果——这次 tap 永远落不下来。保持时线圈
发出蜂鸣意味着 PWM 是可听见的：在线圈两端并联一个 470µF 电容，或把 PWM 提高到 25kHz。
FluidNC 配置从 YAML 接管 `$32`/`$30`，并拒绝实时的 `$` 写入（`error:162`），所以驱动以
`optional=True` 发送它们——一块裸 GRBL 板（它 *确实* 接受这些写入）仍然能正常工作。完整步骤见
[固件页面](/zh/hardware/firmware/)。

## 关于过时片段的提醒

本页取代了旧文档中的时序取值。特别地，**不要** 相信：

- 旧的 `hardware/firmware` 片段 `M3 S1000 / G4 P0.05 (50ms) / M3 S750`——停留时间是
  **`SETTLE_MS = 80ms`**，不是 50ms。
- YAML 注释"S300, tune from S150 up"——保持电平是 **`HOLD_S = S750`**。那些偏低的取值
  会让笔尖松开，把一次 long-press 读成单击。

拿不准时，以这里（以及 `solenoid.py` / `arm.py` 中）的常量为准。
