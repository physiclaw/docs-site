title: MCP tools
description: The 12 tools PhysiClaw exposes to an agent — see the screen, act on a bbox, navigate, enter text, and batch.

PhysiClaw exposes **12 tools** over the Model Context Protocol (FastMCP, streamable-HTTP).
The MCP part is the familiar half: a server names tools, an agent calls them in a loop. The
unfamiliar half is that **every tool here drives a real stylus on a real screen** — there are
no OS touch events, no app APIs. The agent never deals in pixels or motor coordinates. It
names a **bbox** (a rectangle `[left, top, right, bottom]` in `0`–`1` screen fractions) plus an
action; the orchestrator converts the bbox **center** (still `0`–`1`) into GRBL millimetres
through the calibrated affine (`pct_to_grbl`), then drives the arm — `arm.tap` /
`double_tap` / `long_press` / `swipe_to`. So the surprises on this page are *physical*: a
"screenshot" is a gesture apps can see, "swipe up" moves the *stylus* up, and there is genuinely
a hardcoded passcode because the tool phone is a burner.

The tools are grouped by job — **See / Act / Navigate / Text / Sequence** — but listed in
**wire order** (the numbered order the agent receives them) because the most-reached-for tools
(`peek`, `tap`) sit first on purpose. Each tool leads with its one-line job (the first docstring
sentence the agent reads), then parameters, then timing and the safe-to-repeat flag, then the
"what to do next" hint PhysiClaw appends to every result, then any trap.

:::note[Locking and auto-park (true of every action tool)]
Every gesture holds a **busy lock** and rejects calls that overlap a running gesture or that
would cross the AssistiveTouch button. After each gesture the arm **auto-parks off-screen** so
the stylus never blocks the camera. That is why a `peek` right after any action is always a
clean view — and why every action result ends with a hint to do exactly that.
:::

## See

The agent calls one of these *before* an action to ground a target's bbox, and *again after*
to check the screen changed. Both return the same shape — `[Image, listing]` — a JPEG with
icon boxes drawn, plus a plain-text listing, one row per element. **All bboxes everywhere are
`0`–`1` screen fractions, never pixels.**

```text
id [kind] "label" [left,top,right,bottom] conf
```

### 1. `peek()`

> Default view tool — overhead camera (~4s, non-mutating).

- **Parameters:** none.
- **Timing / safety:** ~4s, **non-mutating** — safe to call as often as you like.
- **When:** before any tap/swipe to find the target; after, to verify. An *identical* listing
  across two peeks means the screen didn't change.

:::tip[Reading the result]
`peek` draws the detected boxes onto the JPEG, so the agent can cross-check a bbox visually
against the listing before acting.
:::

### 2. `screenshot()`

> Phone's pixel-perfect capture (~12s, MUTATING) — escalate when peek misses the target.

- **Parameters:** none.
- **Timing / safety:** ~12s, **mutating**.
- **Why mutating:** it fires the **iOS screenshot gesture**, which apps can *observe and react
  to* (share sheets, "similar items" panels, watermarked frames). **Always `peek` after a
  `screenshot` before tapping** — the screen state may have shifted.
- **Use only when** `peek` can't do the job: the target is too small for the camera to resolve,
  glare or motion blur makes the frame unreadable, or there's fine print to read.

`peek` is roughly three times faster and leaves no trace, so it's the default; `screenshot` is
the escalation.

## Act

Every action tool takes a `bbox` and operates at its **center**. The arm drives the stylus
there; the solenoid drops the tip, and the gesture's **duration**
decides whether it reads as a tap, a double-tap, or a hold.

### 3. `tap(bbox)`

> Tap once at the bbox center — buttons, links, list items, dismissing dialogs.

- **Next:** `peek` to verify and ground the next target. An identical listing means the tap
  missed (retry once) or the bbox was wrong (pick a different element from the new peek).

### 4. `double_tap(bbox)`

> Two quick taps (~150ms apart) at the bbox center — for zoom or word-select.

- **Use for** zooming maps / photos / web pages, or selecting a word in editable text.
- For buttons, use `tap`.

### 5. `long_press(bbox)`

> Press and hold ~1.2s at the bbox center — context menus, edit mode, paste popover.

- **Use for** opening context menus, entering icon-rearrange edit mode, or triggering the
  paste popover after `send_to_clipboard`.
- **Next:** always `peek` — you'll need a fresh bbox from the popover that just appeared (Paste
  / Copy / etc.) before you can tap it.

### 6. `swipe(bbox, direction, size="m", speed="medium")`

> Slide the stylus across `bbox` in `direction` (1–8cm) — for scrolling, paging, dismissing
> cards, opening Control / Notification Center.

| Parameter | Values | Notes |
| --- | --- | --- |
| `bbox` | `[l,t,r,b]` in 0–1 | the gesture origin |
| `direction` | `up` `down` `left` `right` | the **stylus** motion, not the page motion |
| `size` | `s` `m` `l` `xl` `xxl` | stroke length, default `m` |
| `speed` | `slow` `medium` `fast` | stylus velocity, default `medium` |

The `size` scale maps to a real stroke length (a fraction of the screen):

| `size` | Screen fraction | Length | Use |
| --- | --- | --- | --- |
| `s` | 0.10 | ≈ 1cm | small nudge |
| `m` | 0.30 | ≈ 2cm | most scrolls (default) |
| `l` | 0.50 | ≈ 4cm | long scroll |
| `xl` | 0.75 | ≈ 6cm | page-sized scroll |
| `xxl` | 0.90 | ≈ 8cm | full-screen (Control / Notification Center) |

:::caution[Direction is the stylus, not the page]
Swiping **up** scrolls the page **down** (reveals content below). Swiping **down** scrolls
**up** (opens Notification Center). Swiping **left** moves to the next photo / page. Faster
strokes also coast further on iOS scroll lists.
:::

- **Next:** `peek` to verify the page scrolled and plan the next move.

## Navigate

These take **no bbox** — each issues a fixed iPhone system gesture.

### 7. `home_screen()`

> Return to the iPhone home screen — exit any app to a known launch pad.

- Issues the swipe-up-from-bottom gesture.
- **Use to** start a fresh task or recover from getting lost in app navigation.
- **Next:** `peek` to plan your next tap on the home-screen icons.

### 8. `go_back()`

> Pop back one screen via the iPhone left-edge swipe gesture.

- Works in apps with a navigation stack (most do). The same screen after `peek` means either
  the gesture didn't register (retry once) or this screen has no back action (modals, root
  tabs, lock screen) — try `home_screen` and re-enter, or tap an in-screen `<` / Back button.

:::caution[Image viewers eat the edge swipe]
Full-screen image viewers (product images, Messages / WeChat photos) reclaim left/right swipes
for image navigation, so the edge-swipe won't pop. Close via the in-viewer `X` / `Done` button
instead.
:::

### 9. `force_quit()`

> Force-quit the current app via the iOS app-switcher gesture (~7s) — hard reset when you're
> stuck on the wrong app page and can't find the entry you need from here.

- **Timing:** ~7s; lands on the home screen.
- **Use after** `go_back` hasn't reached the right entry point: popups won't dismiss, the back
  stack loops, or the wrong page keeps returning.
- **Next:** reopen the app fresh from the home screen.

### 10. `unlock_phone()`

> Unlock the phone with passcode `111111` (~12s).

It wakes the screen, swipes up, waits for Face ID to fail, OCRs the keypad, and taps each digit.
The passcode is **hardcoded to `111111`** — a throwaway/burner tool-phone code, so a real
password never leaks through git or logs.

:::caution[On failure, don't retry — the fix is on the user's side]
Every other tool needs the phone unlocked, so a **failed unlock is terminal for the session**.
The user must set the tool phone's code to `111111`, or disable auto-lock
(Settings → Display & Brightness → Auto-Lock → Never), otherwise taps and unlock fail.
:::

- **Next:** `peek` to confirm you're on the home screen and plan the next tap.

## Text

### 11. `send_to_clipboard(text)`

> Copy `text` to the phone's clipboard — paste is far faster than typing.

The standard flow:

<Steps>

1. `send_to_clipboard(text)`

2. `long_press(field_bbox)` — opens the paste popover.

3. `tap` the `Paste` (or `粘贴`) button that appears.

</Steps>

Fall back to the on-screen keyboard only if the field rejects paste (passcode fields, some
search bars).

## Sequence

### 12. `sequence(step1, step2?, step3?, step4?, step5?)`

> Run up to 5 actions in one call — saves turns on deterministic flows.

Batch up to **5 actions under a single busy lock** when intermediate observations would add
nothing — opening an app via `tap → tap → tap`, or paste + send a message.

Each step is a dict with two fields:

- `tool_name` — one of `tap` / `double_tap` / `long_press` / `swipe` / `send_to_clipboard`.
- `arg` — that tool's argument: a `bbox` for tap/double_tap/long_press,
  `{bbox, direction, size?, speed?}` for swipe, a string for send_to_clipboard.

:::caution[Stops at the first failure — no rollback]
Earlier steps are **not** undone. `peek` after a failure to see where it landed before
retrying. Don't batch a flow when a dialog *may or may not* appear between steps, or when a
target's bbox is uncertain — verify with `peek` first.
:::

- **Next:** `peek` to verify the final state and plan the next move.
