---

PhysiClaw 通过 Model Context Protocol（FastMCP，streamable-HTTP）暴露 **12 个工具**。
其中 MCP 这一半是大家熟悉的：服务端为工具命名，助理在循环里调用它们。陌生的另一半在于
**这里的每个工具都驱动一支真实的触控笔在一块真实的屏幕上操作**——没有操作系统级的触摸事件，
也没有 app API。助理从不直接处理像素或电机坐标。它只指定一个 **bbox**（一个矩形
`[left, top, right, bottom]`，以 `0`–`1` 的屏幕比例表示）外加一个动作；
orchestrator 通过标定好的仿射变换（`pct_to_grbl`）把 bbox 的 **中心**（仍是 `0`–`1`）
换算成 GRBL 毫米坐标，再驱动机械臂——`arm.tap` / `double_tap` / `long_press` / `swipe_to`。
所以本页的意外之处都是 *物理性* 的：一次 "screenshot" 是 app 能看见的手势，"向上 swipe"
移动的是 *触控笔*，而那个硬编码的密码确实存在——因为这台工具手机是一次性的。

工具按职责分组——**看 / 操作 / 导航 / 文本 / 序列**——但按 **传输顺序（wire order）** 列出
（也就是助理收到它们的编号顺序），因为最常用的工具（`peek`、`tap`）被有意排在最前面。
每个工具的开头是它的单行职责（助理读到的第一句 docstring），接着是参数，然后是时序与
可安全重复的标记，再是 PhysiClaw 追加到每次结果末尾的"下一步该做什么"提示，最后是各自的陷阱。

:::note[加锁与自动归位（每个动作工具都成立）]
每个手势都会持有一把 **busy 锁**，拒绝与正在执行的手势重叠、或会越过 AssistiveTouch 按钮的调用。
每个手势之后机械臂会 **自动归位到屏幕外**，这样触控笔就不会挡住相机。这正是为什么任何动作之后
紧跟一次 `peek` 总能得到一帧干净的画面——也是为什么每次动作结果都以这样的提示收尾。
:::

## 看（See）

助理在动作 *之前* 调用其中一个工具来锁定目标的 bbox，并在动作 *之后再次* 调用以确认屏幕已发生变化。
两者返回相同的结构——`[Image, listing]`——一张画出了图标框的 JPEG，外加一段纯文本清单，每个元素一行。
**这里所有的 bbox 都是 `0`–`1` 的屏幕比例，绝不是像素。**

```text
id [kind] "label" [left,top,right,bottom] conf
```

### 1. `peek()`

> 默认视图工具——俯拍相机（约 4 秒，non-mutating）。

- **参数：** 无。
- **时序 / 安全性：** 约 4 秒，**non-mutating**——想调用多少次都安全。
- **时机：** 在任何 tap/swipe 之前用来找到目标；动作之后再用来验证。两次 peek 返回 *完全相同* 的清单，
  说明屏幕没有变化。

:::tip[读取结果]
`peek` 会把检测到的框直接画到 JPEG 上，这样助理在操作前可以把 bbox 与清单做可视化交叉核对。
清单是如何生成的，参见 [它如何看见](/zh/concepts/how-it-sees/)。
:::

### 2. `screenshot()`

> 手机自身的像素级精确截图（约 12 秒，MUTATING）——当 peek 找不到目标时升级使用。

- **参数：** 无。
- **时序 / 安全性：** 约 12 秒，**mutating**。
- **为什么会改变状态：** 它会触发 **iOS 的截图手势**，而 app *可以观察并对其作出反应*
  （分享面板、"相似商品"面板、带水印的画面）。**在 `screenshot` 之后、tap 之前务必先 `peek`**——
  屏幕状态可能已经变了。
- **仅在** `peek` 无法胜任时使用：目标太小相机无法分辨、眩光或运动模糊导致画面无法读取、
  或者有需要阅读的小字。

`peek` 速度大约快三倍且不留痕迹，所以它是默认选择；`screenshot` 是升级方案。
为什么要这样划分，详见 [它如何看见](/zh/concepts/how-it-sees/)。

## 操作（Act）

每个动作工具都接收一个 `bbox`，并在其 **中心** 处操作。机械臂把触控笔驱动到那里；
[电磁铁（solenoid）](/zh/reference/gestures/) 让笔尖落下，而手势的 **持续时长** 决定它被识别为
单击、双击还是长按（电磁铁的标准时序见 [手势](/zh/reference/gestures/)）。

### 3. `tap(bbox)`

> 在 bbox 中心单击一次——按钮、链接、列表项、关闭对话框。

- **下一步：** `peek` 来验证并锁定下一个目标。清单完全相同意味着 tap 没点中（重试一次），
  或者 bbox 错了（从新的 peek 中挑选另一个元素）。

### 4. `double_tap(bbox)`

> 在 bbox 中心快速点两下（间隔约 150ms）——用于缩放或选词。

- **用于** 缩放地图 / 照片 / 网页，或在可编辑文本中选中一个词。
- 对于按钮，请用 `tap`。

### 5. `long_press(bbox)`

> 在 bbox 中心按住约 1.2 秒——上下文菜单、编辑模式、粘贴弹窗。

- **用于** 打开上下文菜单、进入图标重排的编辑模式，或在 `send_to_clipboard` 之后触发粘贴弹窗。
- **下一步：** 始终 `peek`——你需要从刚出现的弹窗中（Paste / Copy 等）获取一个新的 bbox 才能点击它。

### 6. `swipe(bbox, direction, size="m", speed="medium")`

> 让触控笔沿 `direction` 方向滑过 `bbox`（1–8cm）——用于滚动、翻页、划掉卡片、打开控制中心 / 通知中心。

| 参数 | 取值 | 说明 |
| --- | --- | --- |
| `bbox` | `[l,t,r,b]`，0–1 区间 | 手势起点 |
| `direction` | `up` `down` `left` `right` | **触控笔** 的运动方向，不是页面的运动方向 |
| `size` | `s` `m` `l` `xl` `xxl` | 滑动行程长度，默认 `m` |
| `speed` | `slow` `medium` `fast` | 触控笔速度，默认 `medium` |

`size` 等级对应真实的行程长度（屏幕的一个比例）：

| `size` | 屏幕比例 | 长度 | 用途 |
| --- | --- | --- | --- |
| `s` | 0.10 | ≈ 1cm | 小幅轻推 |
| `m` | 0.30 | ≈ 2cm | 大多数滚动（默认） |
| `l` | 0.50 | ≈ 4cm | 长滚动 |
| `xl` | 0.75 | ≈ 6cm | 整页式滚动 |
| `xxl` | 0.90 | ≈ 8cm | 全屏（控制中心 / 通知中心） |

:::caution[方向指的是触控笔，不是页面]
向 **上** swipe 会让页面 **向下** 滚动（显示下方内容）。向 **下** swipe 会让页面 **向上** 滚动
（打开通知中心）。向 **左** swipe 切到下一张照片 / 下一页。在 iOS 滚动列表里，更快的滑动还会惯性滑得更远。
:::

- **下一步：** `peek` 验证页面是否已滚动并规划下一步。

## 导航（Navigate）

这些工具 **不接收 bbox**——每个都发出一个固定的 iPhone 系统手势。

### 7. `home_screen()`

> 返回 iPhone 主屏幕——从任意 app 退出到一个已知的起跳点。

- 发出从底部上滑的手势。
- **用于** 开始一个新任务，或从 app 导航中迷路时恢复。
- **下一步：** `peek` 来规划你在主屏幕图标上的下一次 tap。

### 8. `go_back()`

> 通过 iPhone 的左边缘滑动手势返回上一屏。

- 在带有导航栈的 app 中有效（大多数 app 都有）。`peek` 后屏幕仍相同，说明要么手势没被识别（重试一次），
  要么此屏幕没有返回动作（模态框、根标签页、锁屏）——试试 `home_screen` 后重新进入，
  或点击屏幕内的 `<` / 返回按钮。

:::caution[图片查看器会吞掉边缘滑动]
全屏图片查看器（商品图、Messages / WeChat 照片）会把左右滑动占用为图片切换，所以边缘滑动不会返回。
改用查看器内的 `X` / `Done` 按钮关闭。
:::

### 9. `force_quit()`

> 通过 iOS app 切换器手势强制退出当前 app（约 7 秒）——当你卡在错误的 app 页面、又无法从这里找到所需入口时，做一次硬重置。

- **时序：** 约 7 秒；落到主屏幕。
- **用于** `go_back` 无法到达正确入口点之后：弹窗无法关闭、返回栈循环往复，或者总是返回错误的页面。
- **下一步：** 从主屏幕重新打开 app。

### 10. `unlock_phone()`

> 用密码 `111111` 解锁手机（约 12 秒）。

它会唤醒屏幕、上滑、等待 Face ID 失败、OCR 识别键盘，然后逐位点击数字。密码 **硬编码为 `111111`**——
这是一个一次性 / 工具手机的密码，所以真实密码绝不会通过 git 或日志泄露。

:::caution[失败后不要重试——修复方法在用户一侧]
其余每个工具都需要手机处于解锁状态，所以一次 **解锁失败会终结整个会话**。
用户必须把工具手机的密码设为 `111111`，或者关闭自动锁定
（设置 → 显示与亮度 → 自动锁定 → 永不），否则 tap 和解锁都会失败。
:::

- **下一步：** `peek` 确认你在主屏幕上并规划下一次 tap。

## 文本（Text）

### 11. `send_to_clipboard(text)`

> 把 `text` 复制到手机剪贴板——粘贴远比逐键敲字快。

标准流程：

<Steps>

1. `send_to_clipboard(text)`

2. `long_press(field_bbox)`——打开粘贴弹窗。

3. `tap` 出现的 `Paste`（或 `粘贴`）按钮。

</Steps>

只有在输入框拒绝粘贴时（密码输入框、某些搜索栏）才退回到屏幕键盘。

## 序列（Sequence）

### 12. `sequence(step1, step2?, step3?, step4?, step5?)`

> 在一次调用中运行最多 5 个动作——在确定性流程上节省回合数。

当中间的观察不会带来任何价值时，可以把最多 **5 个动作批量放进同一把 busy 锁** 下——
比如通过 `tap → tap → tap` 打开一个 app，或者粘贴 + 发送一条消息。

每一步都是一个含两个字段的字典：

- `tool_name`——`tap` / `double_tap` / `long_press` / `swipe` / `send_to_clipboard` 之一。
- `arg`——该工具的参数：tap/double_tap/long_press 用 `bbox`，swipe 用
  `{bbox, direction, size?, speed?}`，send_to_clipboard 用字符串。

:::caution[遇到第一个失败即停止——不回滚]
之前的步骤 **不会** 被撤销。失败后先 `peek` 看看停在哪里，再重试。当步骤之间 *可能出现也可能不出现*
对话框，或目标的 bbox 不确定时，不要把流程批量执行——先用 `peek` 验证。
:::

- **下一步：** `peek` 验证最终状态并规划下一步。
