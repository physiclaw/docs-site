这里，是"我提示它"变成"它自己跑起来"的地方。内置
[agent](/zh/agent/overview/) 不等你来问 —— **一个 trigger 把它唤醒，它操作手机，然后睡去。**

## 它是怎么醒来的

server 运行时，一个运行时循环（`agent/runtime/runtime.py`）以固定间隔滴答（默认 **1.0s**），
挨个问每个 *hook* 一个问题：有事发生吗？没事？它就睡去，下一次再滴答。当某个 hook 报告了一个 trigger，
循环就唤醒一个全新的**会话**去处理它，然后回到滴答。

```text
 tick ──► hook: "anything happen?" ──► no ──► sleep ──► tick …
                       │
                       └─ yes ──► wake a session ──► act ──► cooldown ──► tick …
```

有两道闸门，会让 agent 一直睡到设备真正就绪：

- **只有当 `/api/status` 报告 `ready=true` 时，循环才会滴答。** 这个标志会被
  [校准](/zh/set-up/calibrate/)的最后一步翻为 true。在那之前，server 可以是开着的、你也仍能手动操控它，
  但 agent 保持沉睡。
- **必须配置好一个模型。** 即便在 `ready=true` 下，没有模型 agent 也不会醒 —— 用
  `physiclaw models key <provider>` 然后 `physiclaw models use <provider/model>`（或 `PHYSICLAW_MODEL`）来设置它。
  一个没有模型的 server 仍可用于硬件设置和手动 MCP 调用，它只是永远不会做出反应。参见 [模型](/zh/agent/models/)。

## 两种 trigger

每一次滴答都会轮询两个 hook。一个盯着时钟，另一个盯着屏幕：

<CardGrid>
  <Card title="Cron —— 按计划" icon="seti:clock">
    cron hook（`agent/hooks/cron.py`）每次滴答都读 `~/.physiclaw/jobs/jobs.md`，触发任何下次触发时刻已到、
    且状态为 `pend` 的作业。这是周期性工作："早上 8 点查天气"、"每周五下生鲜单"。
  </Card>
  <Card title="Poll —— 屏幕变化时" icon="seti:image">
    poll hook（`agent/hooks/poll.py`）调用 `GET /api/phone/watch` —— 头顶的摄像头实实在在地盯着手机屏幕，
    画面一变就触发：来了新消息、来了通知、等的那个画面出现了。这是反应，而不是按表运行。
  </Card>
</CardGrid>

## 一次只跑一个会话

循环绝不同时跑两个会话。检查 hook 和运行反应是*按序 await* 的，所以不会有新的滴答在反应进行到一半时启动。
一次反应结束后，循环会睡 `react_cooldown_seconds`（默认 **6.0s**，`CONFIG.engine.react_cooldown_seconds`）——
足够长，让屏幕动画安定下来、让下一次 watch 轮询在它再次检查前重新建立基线。agent 不会踩到自己刚做的触控笔动作。

## 什么是一个"会话"

一个会话，是 agent 的看 → 决策 → 动作循环一次自成一体的运行，为处理刚刚触发的那些 trigger 而启动。
每个会话都从头开始 —— 全新的对话、自己的轨迹 —— 但它并不健忘：它在醒来时加载
[持久记忆](/zh/agent/memory-and-skills/)，而且被告知当前日期、以及究竟是哪些 trigger 唤醒了它。

在底层，反应会以两种方式之一被分发：对 `claude-code/<model>` 这种 agent，它会把 Anthropic 的 `claude` CLI
作为子进程运行（`spawn_claude`）；对其他任何 provider，它则运行进程内引擎。无论哪种方式，上面那个循环都是一样的 ——
内部细节参见 [引擎内部](/zh/agent/inside-the-engine/)。

一个会话一直运行，直到 agent 用一个单词**裁决**关闭它：

- **`DONE`** —— 任务完成了。
- **`WAIT`** —— 工作还没做完；稍后回来接着干（下文详述）。
- **`FAIL`** —— 它没法完成，也不会再去做了。
- **`IDLE`** —— 没什么可做的；这次唤醒不需要动作（例如一次误报的手机 watch）。
- **`STUCK`** —— 它耗尽了回合数、provider 撑不住了，或者撞上了一个无法恢复的拦路石。PhysiClaw 会重试一个 `STUCK`
  会话几次，才接受它。（`DONE` / `IDLE` / `WAIT` / `FAIL` 都干净退出、不重试。）

## 排定一个周期性任务

排定的工作住在一个人类可读的文件 `~/.physiclaw/jobs/jobs.md` 里。每个 `## <id>` 小节就是一个作业。
**不要手动编辑触发时刻** —— 加作业最简单的办法是叫 agent 来做，它会用内置的 `cron` 技能写一个格式正确的条目，
并重新验证文件依然能被解析。

一个作业长这样：

```markdown
## morning-weather

Tell me the day's forecast every morning.

- Type: periodic
- Schedule: `0 8 * * *`
- Create time: 2026-06-22T21:40
- Next fire time: 2026-06-23T08:00
- Last fire time: (never)
- Execution time: (never)
- Execution result: (never)
- Status: pend
- Context: Open the weather app, read today's high/low and conditions, and message them to me on WeChat.
```

有三个字段承载着含义：

- **`Schedule`** 是标准的 5 字段 cron —— `min hour day-of-month month day-of-week` ——
  支持 `*`、`N`、`N,M`、`*/N` 和 `A-B`（day-of-week 中 `0`=周日…`6`=周六）。`0 8 * * *`
  是"每天 08:00"；`*/30 * * * *` 是"每 30 分钟"。
- **`Context`** 是这次运行交给 agent 的任务说明 —— 开哪个 app、要做什么、有哪些边界情况。（标题下那一行描述是给
  *你*翻文件时看的；`Context` 才是给干活的 agent 看的。）
- **`Type`** 取 `periodic`（每次计划匹配就触发）或 `one-time`（触发一次，然后彻底关闭）。
  对于后续跟进、提醒、和延后的动作，`one-time` 是默认值。

:::note[作业从不被删除]
作业是在状态间流转，而不是凭空消失：`pend` → `fired`（hook 触发了它）→ 对周期作业回到 `pend`，
或对一次性作业变为 `done` / `fail`。你把作业的状态设为 `cancel` 来取消它。终结状态的作业（`cancel` / `done` / `fail`）
会在 7 天后自动清除。agent 通过 cron CLI 自己标记完成 ——
`python -m physiclaw.agent.hooks.cron done|fail|cancel <id> [result summary]`（运行时会以
`uv run python -m physiclaw.agent.hooks.cron …` 的形式调用它）。
:::

## "稍后回来看" —— WAIT 裁决

有些任务没法一遍走完：你下了一单，得等一个确认；或者你在盯着一个还没来的回复。这正是 `WAIT`
的用处 —— agent 在结束会话时已决定稍后接着干。

**通常 agent 会在关闭前自己排好后续** —— 它写一个 `one-time` 作业，定在它想醒来的那个确切时刻。这是正确的做法，
因为 agent 知道下一个状态大概什么时候会就绪。

**兜底机制：** 要是它带着 `WAIT` 关闭却*忘了*排一个，PhysiClaw 会自动创建一个单独的通用后续作业（规范 id
`wait-check-auto`，`agent/engine/jobs.py`），它在 **约 15 分钟**后触发、重新检查状态、接着干。这个通用的 15 分钟唤醒
对具体任务来说通常是错的 —— 这正是为什么要让 agent 自己去排。同一个规范 id 会在多次 WAIT 间复用，
所以这个文件不会每暂停一次就多长出一条新条目。

结果就是：一桩长跑的差事能跨过多次睡眠存活下来。agent 当下能做的就做，把一次唤醒停在稍后，下一个会话再把这根线头接回来
—— 且记忆完好无损。

## 安全地管理作业

`jobs.md` 是纯 Markdown，但手动编辑触发时刻或状态字段，是把它弄坏的捷径。有两条安全的路：

- **叫 agent 来做。** 内置的 `cron` 技能会在每次改动前后都验证文件能被解析，所以一处格式错误的编辑混不进去。
- **用 cron CLI。** 你可以用 cron 技能的 `verify` 命令查看文件，而 `done|fail|cancel <id>` 这几个子命令
  无需碰原始字段就能管理一个作业的生命周期。

完整的命令清单参见 [CLI 参考](/zh/reference/cli/)，agent 在这些无人值守的运行之间带着什么，则参见
[记忆与技能](/zh/agent/memory-and-skills/)。
