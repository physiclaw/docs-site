到目前为止，你一直是从外部驱动 PhysiClaw：由一个 MCP 客户端来挑选方框，PhysiClaw
负责伸出那根手指。而这一节要回答的，正是你大概早就在问的问题 —— **PhysiClaw
非得靠一个外部 agent 来驱动吗，还是它能自己驱动自己？** 两种都行。同一套安装有两种跑法，
而你只需改配置就能切换，从不需要重装。

- **纯 MCP server。** 一个外部客户端 —— Claude Desktop、某个 IDE、OpenClaw、任何 MCP
  客户端 —— 连上 `http://<host>:8048/mcp`，透过 FastMCP 接收 PhysiClaw 的指令，由它做全部
  决策。PhysiClaw 只负责暴露那[十二个工具](/zh/reference/mcp-tools/)，叫它点哪儿它就点哪儿。
  这是被动模式：在客户端发来工具调用之前，它一直闲着。
- **内置 agent。** PhysiClaw 自己就是那个大脑。它在自己的进程里跑自己的看 → 决策 → 动作
  循环，不外接任何客户端。同样的机器人、同样的十二个工具 —— 区别只在于，是谁的脑子在掌舵。

这一页讲的是内置 agent：它多带来了什么、它大致怎么思考、它并不被锁死在某一个模型上，以及什么时候该用哪一种。
更深入的细节由后面的页面来讲。

:::tip[从 OpenClaw 过来的？]
OpenClaw 是透过它所栖身的屏幕来驱动软件的。PhysiClaw 的内置 agent 遵循同一套打法 ——
一个原生工具调用循环，在 provider 边界处归一化 —— 只不过它瞄准的是一台*实体*手机。同样的原则，
但装上了身体：那个曾经点击像素的 agent，如今伸出手，敲在玻璃上。
:::

## 内置 agent 多带来了什么

纯 MCP server 是**被动的** —— 它等着外部客户端来动手。而内置 agent 可以是**发起者**。
它具体带来了四样东西：

<CardGrid>
  <Card title="它掌控循环" icon="seti:default">
    agent 自己跑完整个 `peek` → 挑一个 bbox 和一个手势 → `tap` → 再 `peek` 的循环。
    循环里没有外部模型 —— 是 PhysiClaw 自己在挑方框。
  </Card>
  <Card title="它无人值守运行" icon="moon">
    它自己醒来 —— 按计划（cron 任务，位于 `~/.physiclaw/jobs/jobs.md`），或当手机屏幕发生
    变化时（`phone_watch` 的相机唤醒轮询）—— 操作一番，然后睡去。没人需要守在客户端前。
    见[自主任务](/zh/agent/autonomous-tasks/)。
  </Card>
  <Card title="它会记事" icon="open-book">
    一份持久的记忆位于 `~/.physiclaw/memory`，把事实带过每一次醒来，所以 agent 不必每回
    都从零开始。见[记忆与技能](/zh/agent/memory-and-skills/)。
  </Card>
  <Card title="它遵循技能" icon="puzzle">
    **技能**是可复用、针对具体 app 的操作手册，位于 `~/.physiclaw/skills` ——
    "怎么发一条微信消息"、"怎么下一单生鲜" —— agent 会去发现并遵循它们，而不必每次都把一个
    app 重新摸索一遍。见[记忆与技能](/zh/agent/memory-and-skills/)。
  </Card>
</CardGrid>

## agent 怎么思考

每次醒来都跑你在[工作原理](/zh/start/how-it-works/)里见过的那个循环 ——
只不过这回是 PhysiClaw 在驱动它，而非外部客户端：

```text
 wake ──► LOOK ──► DECIDE ──► ACT ──► LOOK ──► … ──► close
 trigger  peek     pick a      tap /   peek      with a short
 fires    (camera) bbox +      swipe   again,    verdict
                   gesture             re-decide
```

关键的习惯：每个回合**总是以查看结果收尾**。agent 走一步动作，然后再 `peek` 一眼看看变了什么 ——
于是一个弹窗、一次缓慢加载、一个错误的屏幕，不过是下一个要去应对的状态，而不会让脚本脱轨。
agent 绝不盲目地连点一通。等活儿干完，它会自己关掉会话，并附上一句简短的裁决，讲清发生了什么 ——
任务完成、被卡住、暂停等回复，或者无事可做。

确切的回合结构（一条滚动记录加一个动作）、那些裁决词，以及一次长时间的运行如何待在上下文窗口
之内，都属于引擎内部细节 —— [引擎内部](/zh/agent/inside-the-engine/)会讲到这些。

## 一个大脑，多种模型

内置 agent 不绑定任何一家厂商。你用一行 `provider/model` 形式的配置引用来选定模型，
而这个引用决定了跑哪个引擎：

- 形如 `claude-code/<model>` 的引用，会以子进程方式调起 Anthropic 的 `claude` CLI。
- **其他任何** provider id，都走 PhysiClaw 的进程内引擎，它能对接六家 provider：`qwen`
  （DashScope）、`moonshot`（Kimi）、`openai`、`anthropic`（直连 `/v1/messages` API）、
  `google`（Gemini 的 OpenAI 兼容端点）和 `deepseek`。

:::note[`anthropic` 和 `claude-code` 不是同一个 provider]
`anthropic` 是直接调用 Anthropic `/v1/messages` API 的进程内 provider。`claude-code`
则是另一种引擎形态，它以子进程方式运行 `claude` CLI。两者共用 Anthropic 的密钥和模型目录，
但是两个不同的选项。[模型](/zh/agent/models/)有完整列表和切换方法。
:::

:::caution[没设模型，agent 就不会醒]
**没有默认模型。** 在你配置一个模型之前，内置 agent 一直沉睡：

```bash
physiclaw models key <provider>          # store the API key
physiclaw models use <provider/model>    # select the active model
```

（或设置 `PHYSICLAW_MODEL`。）一个没设模型的纯 `physiclaw server` **照样能跑** ——
用于硬件设置和手动 MCP 调用 —— 但内置 agent 不会启动。如果 PhysiClaw "什么都不干"，
这是第一个该检查的地方。见[模型](/zh/agent/models/)。
:::

## 何时用哪种

你不必一次性做出永久选择 —— 同一套安装两种都支持。

| | 纯 MCP server | 内置 agent |
| --- | --- | --- |
| 谁来决策 | 你的外部客户端（Claude Desktop、某个 IDE、OpenClaw） | PhysiClaw 自己 |
| 谁来发起任务 | 你，靠对客户端发提示 | 一个触发器：一份计划，或一次屏幕变化 |
| 无人值守运行 | 否 —— 需要连着一个客户端 | **是** —— 醒来、动作、睡去 |
| 记忆与技能 | 取决于你的客户端 | 内置 |

想让一个现有的 agent 继续掌舵、只给它装上手，就用**纯 server**。想让 PhysiClaw 自己跑起来 ——
一桩周期性的杂务、一个守着等着就动手的任务、一台在你不在时也照样做事的手机 —— 就用**内置 agent**。

从这里出发，[模型](/zh/agent/models/)给 agent 配上一个大脑，接着
[自主任务](/zh/agent/autonomous-tasks/)给它装上触发器，
[记忆与技能](/zh/agent/memory-and-skills/)给它一份记忆。
