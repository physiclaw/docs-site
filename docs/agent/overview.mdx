You've been driving PhysiClaw from the outside: an MCP client picks the boxes and
PhysiClaw supplies the finger. The question this section answers is the one you're
probably already asking — **does PhysiClaw need an outside agent to drive it, or can it
drive itself?** It can do either. The same install runs two ways, and you switch with
config, never a reinstall.

- **Plain MCP server.** An external client — Claude Desktop, an IDE, OpenClaw, any MCP
  client — connects to `http://<host>:8048/mcp`, receives PhysiClaw's instructions over
  FastMCP, and does all the deciding. PhysiClaw just exposes the
  [twelve tools](/en/reference/mcp-tools/) and taps where it's told. This is the reactive
  mode: it sits idle until a client sends a tool call.
- **Built-in agent.** PhysiClaw itself is the brain. It runs its own look → decide → act
  loop, in its own process, with no external client attached. Same robot, same twelve
  tools — the only difference is whose mind is in charge.

You met the plain server in [Operating modes](/en/run/operating-modes/). This page is the
map for the other half: what the built-in agent adds, roughly how it thinks, that it isn't
locked to one model, and when to reach for which. The deeper pages handle the details.

:::tip[Coming from OpenClaw?]
OpenClaw drives software through the screen it lives on. PhysiClaw's built-in agent follows
the same playbook — a native tool-call loop, normalized at the provider boundary — but
points it at a *physical* phone. Same principles, with a body: the agent that used to click
pixels now reaches out and taps glass.
:::

## What the built-in agent adds

A plain MCP server is **reactive** — it waits for an external client to act. The built-in
agent can be the **initiator**. Four concrete things come with it:

<CardGrid>
  <Card title="It owns the loop" icon="seti:default">
    The agent runs the full `peek` → pick a bbox and a gesture → `tap` → `peek`-again cycle
    itself. No external model in the loop — PhysiClaw is choosing the boxes.
  </Card>
  <Card title="It runs unattended" icon="moon">
    It wakes on its own — on a schedule (cron jobs in `~/.physiclaw/jobs/jobs.md`) or when
    the phone screen changes (the `phone_watch` camera-wake poll) — acts, then sleeps. No
    one has to be sitting at a client. See [Autonomous tasks](/en/agent/autonomous-tasks/).
  </Card>
  <Card title="It remembers" icon="open-book">
    A persistent memory under `~/.physiclaw/memory` carries facts across wakes, so the
    agent isn't starting cold every time. See [Memory &amp; skills](/en/agent/memory-and-skills/).
  </Card>
  <Card title="It follows skills" icon="puzzle">
    **Skills** are reusable, app-specific playbooks under `~/.physiclaw/skills` — "how to
    send a WeChat message," "how to place a grocery order" — that the agent discovers and
    follows instead of re-figuring-out each app. See [Memory &amp; skills](/en/agent/memory-and-skills/).
  </Card>
</CardGrid>

## How the agent thinks

Each wake runs the same loop you met in [How it works](/en/start/how-it-works/) — PhysiClaw
just drives it instead of an external client:

```text
 wake ──► LOOK ──► DECIDE ──► ACT ──► LOOK ──► … ──► close
 trigger  peek     pick a      tap /   peek      with a short
 fires    (camera) bbox +      swipe   again,    verdict
                   gesture             re-decide
```

The key habit: a turn **always ends by looking at the result**. The agent takes one action,
then `peek`s again to see what changed — so a popup, a slow load, or a wrong screen is just
the next state to react to, not something that derails a script. The agent never fires a
burst of blind taps. When the work is over, it closes the session itself with a short
verdict that says what happened — task done, blocked, paused for a reply, or nothing to do.

The exact turn structure (one running note plus one action), the verdict words, and how a
long run stays inside the context window are all engine internals —
[Inside the engine](/en/agent/inside-the-engine/) covers them.

## One brain, many models

The built-in agent isn't tied to one vendor. You pick the model with a single config ref in
the shape `provider/model`, and that ref decides which engine runs:

- A ref of `claude-code/<model>` shells out to Anthropic's `claude` CLI as a subprocess.
- **Any other** provider id uses PhysiClaw's in-process engine, which speaks to six
  providers: `qwen` (DashScope), `moonshot` (Kimi), `openai`, `anthropic` (the direct
  `/v1/messages` API), `google` (Gemini's OpenAI-compatible endpoint), and `deepseek`.

:::note[`anthropic` and `claude-code` are not the same provider]
`anthropic` is the in-process provider that calls Anthropic's `/v1/messages` API directly.
`claude-code` is the separate engine flavor that runs the `claude` CLI as a subprocess.
They share Anthropic's key and catalog but are distinct picks. [Models](/en/agent/models/)
has the full list and how to switch.
:::

:::caution[The agent won't wake until a model is set]
There is **no default model**. The built-in agent stays asleep until you configure one:

```bash
physiclaw models key <provider>          # store the API key
physiclaw models use <provider/model>    # select the active model
```

(or set `PHYSICLAW_MODEL`). A plain `physiclaw server` with no model **still runs** — for
hardware setup and manual MCP calls — but the built-in agent won't fire. If PhysiClaw
"isn't doing anything," this is the first thing to check. See [Models](/en/agent/models/).
:::

## When to use which

You don't choose once and for all — the same install does both.

| | Plain MCP server | Built-in agent |
| --- | --- | --- |
| Who decides | your external client (Claude Desktop, an IDE, OpenClaw) | PhysiClaw itself |
| Starts a task | you, by prompting the client | a trigger: a schedule or a screen change |
| Runs unattended | no — needs a client connected | **yes** — wakes, acts, sleeps |
| Memory &amp; skills | up to your client | built in |

Reach for the **plain server** when you want to keep an existing agent in the loop and just
give it hands. Reach for the **built-in agent** when you want PhysiClaw to run on its own —
a recurring chore, a watch-and-react task, a phone doing things while you're away.

From here, [Models](/en/agent/models/) gets the agent a brain, then
[Autonomous tasks](/en/agent/autonomous-tasks/) gives it triggers and
[Memory &amp; skills](/en/agent/memory-and-skills/) gives it a memory.
