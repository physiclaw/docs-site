内置助理在两次醒来之间会忘掉一切 —— 而记忆与技能，就是让它假装没忘的办法。

每个会话都是一次 trigger → 干活 → 退出（见 [自主任务](/zh/agent/autonomous-tasks/)）。模型脑子里的东西没有一样能活到下一次醒来。连续性完全由 `~/.physiclaw` 下的纯 Markdown 文件重建。**记忆是助理知道的，技能是它能做的。** 两者都是你能读、能改、也能手动喂料的文件。

## 记忆：什么会跨会话带走

一个会话就是一次醒来 —— 助理因一个 trigger 而活过来、做完任务、退出。能活下来的，是 `~/.physiclaw/memory/` 下的三个文件：

<FileTree>
- ~/.physiclaw/memory/
  - USER.md          你的档案 —— 由你打理，对助理只读
  - memory.md        持久事实，每行一条
  - 2026-06-22.md    当天的活动日志，每行 `[HH:MM] …`
  - 2026-06-21.md
  - …
</FileTree>

每个文件各司其职，谁来写也各有一条规矩：

| 文件 | 装的是 | 谁来写 |
| --- | --- | --- |
| `USER.md` | 你是谁、你的偏好、你的硬性规矩 | **只有你** —— 助理读它，从不写它 |
| `memory.md` | 持久事实，每行一条 | 助理（用 `save_memory` 追加、就地编辑） |
| `YYYY-MM-DD.md` | 一天一个只追加的文件，`[HH:MM] app: page → page — what it did` | 助理（用 `append_log` 追加） |

### 哪些总是加载，哪些按需拉取

每次醒来，`memory.md` 和你的 `USER.md` 都会被**自动注入系统提示，因为它们小而扎实**（`memory.md` 出现在 `## memory.md` 标题下）。每日日志可能变得很大，所以只有最近的若干条会一起带上：`bootstrap_log_entries` 默认 **10**，作为一份合成的 `read_logs` 结果注入。

当某个任务需要更早的历史时，助理再按需拉取更深的窗口：

```text
read_logs(entries=N)   # default 20 (default_log_entries), max 200
```

这三个数 —— 启动预载、按需默认值、上限 —— 都可以在 `config.toml` 的 `[memory]` 下调。

:::note[为什么把"事实"和"日志"分开？]
`memory.md` 回答*"什么是真的？"* —— 你的地址、你偏好燕麦奶、你买的尺码。每日日志回答*"发生了什么？"* —— `[14:02] jd: cart → checkout — added 2× oat milk ¥39`。事实是无关时间且寥寥几条的，所以总被加载；日志是一卷滚动的磁带且数量众多，所以按需召回。把它们分开，意味着总被加载的那部分保持极小，而日志可以增长却不撑大每一次提示。

正是这种逐步日志，让*未来*的一次醒来能恢复一个半完成的物理任务：它读日志、看见购物车里已经有两件商品，从那儿接着干 —— 正是 [自主任务](/zh/agent/autonomous-tasks/) 里那套逐次醒来、无状态的模型。
:::

### 助理在实践中怎么维护记忆

这些你都不用操心 —— 助理一边干活一边自己维护记忆：

- **你说"记住 X"** → 它用 `save_memory` 往 `memory.md` 追加一行。
- **你改了一个偏好** → 它就地编辑对应的那一行。如果找不到旧文本、或匹配到不止一处，这次编辑会报错，所以事实始终保持每行一条、措辞唯一。
- **它完成了一个有意义的步骤** → 它用 `append_log` 追加一行日志，形如
  `[HH:MM] app: page → page — what it did`。

`USER.md` 是例外 —— 助理只会**读**它。那个文件归你打理。

:::note[两套引擎看的是同一批文件]
无论哪种驱动模式在跑，读的都是同一个 memory 目录。进程内引擎通过 `memory.py` 访问它；`claude-code` 子进程通过它的 `CLAUDE.md` 访问，那里指向的是像 `memory/memory.md` 这样的**相对**路径（不是绝对路径）。两套引擎见 [助理总览](/zh/agent/overview/)。
:::

:::tip[在第一个真任务之前先给 `USER.md` 喂料]
把你的要紧事项一上来就写进 `USER.md` —— 姓名、地址、你用哪些 app、硬性规矩（"绝不从 X 下单"）。它每次醒来都被读到，所以一份好的档案能省得助理反复追问、也省得它猜错。你也可以预先给 `memory.md` 喂料，但要保持小而精。
:::

## 技能：教助理新本领

记忆是助理*知道的*。一个**技能**是它*能做的* —— 一个自成一体的文件夹，教它某个 app 或某类任务的确切工作流。开箱即用时它能点屏幕；一个技能把"点屏幕"变成"在**这个**特定 app 上、用对的方式下一单"。

一个技能就是一个根目录下有 `SKILL.md` 的目录：

<FileTree>
- ~/.physiclaw/skills/
  - jobs/
    - SKILL.md        frontmatter（name + description）+ 工作流正文
  - jd/
    - SKILL.md
    - references/     深层细节，只在需要时加载
    - assets/         参考资料可以指向的自由格式文件
</FileTree>

会扫描两个位置：`~/.physiclaw/skills/`（主目录，这样装好的助理不用 checkout 仓库就能找到它的技能）和相对于运行位置的 `./skills/`（你迭代时很方便）。要是两处都定义了同名技能，**主目录那份胜出**。

### 一个技能怎么被发现、被加载 —— 三个层级

技能是**渐进式**加载的，这样一台装满技能的手机不会淹没提示。助理只为它真正打开的细节买单：

<Steps>

1. **元数据** —— 醒来时，每个技能的 `name` 和一行 `description`
   被注入系统提示。那行描述是助理*唯一*能用来判断一个技能是否相关的信号，正因如此，描述要写成具体的触发条件 —— *"何时用…；不要用于…"* —— 而不是含糊的简介。

2. **工作流** —— 当一个任务匹配上，助理调用 `Skill(name="jd")`
   拿回完整的 `SKILL.md` 正文：那个 app 的章法 —— 它的搜索框、它的结账流程、它的坑。

3. **参考资料** —— 对于罕见的细节，正文会指向一个参考文件，助理调用
   `Skill(name="jd", reference="checkout.md")` 按需把它拉进来。大多数任务永远到不了第 3 层。

</Steps>

这套分层，正是为什么一个搭建者能不停加技能而不拖慢助理。装十个技能，在提示里只花十行单行描述 —— 而不是十套完整工作流。

:::caution[描述是承重的]
描述写得弱的技能，哪怕它正是对的工具也会闲置着，因为助理永远看不到打开它的理由。当你写或装一个技能时，把功夫花在描述上：点名那些该触发它的 app、动词、说法 —— **以及**那些*不*该触发它的情形。
:::

### 例子：`jobs` 技能

内置的 `jobs` 技能是个好范本。它的活儿是排定未来的工作，它的描述精确地点名了何时该用它：

```text
Use when the task involves scheduling future work — any "remind me at …",
"every weekday …", "check again in 30 min", or closing a fired cron job.
… NOT for one-off in-session waits. NEVER edit jobs.md by hand.
```

正文接着教那套工作流：排定的工作住在一个严格、机器解析的 `jobs.md` 里，所以这个技能禁止直接编辑那个文件，把每一处改动都引到一个 CLI 上。助理读到这些，学会安全地排一个提醒，从此再不必去逆向那个文件格式。这就是每个技能的套路：把一件事的正确做法编码一次，往后每个会话都继承它。

正是这同一个技能，驱动了 [自主任务](/zh/agent/autonomous-tasks/) 里讲的那些 cron 和 poll trigger。

### 安装、列出、移除技能

技能来自一个外部 git 仓库（一个顶层有 `skills/<name>/SKILL.md` 的仓库）。CLI 克隆来源、把技能目录树拷进 `~/.physiclaw/skills/<name>/`：

```bash
physiclaw skills install <name> [--from owner/repo|url|path] [--ref <ref>] [--force]
physiclaw skills list
physiclaw skills uninstall <name>
```

有几条规矩值得知道：

- **`--from` 的解析：** 单次调用的 `--from` 旗标胜过配置里的 `default_source`。解析结果落在 `~/.physiclaw` **内部**的来源会被拒绝 —— 技能必须来自外部仓库，而不是你自己的数据目录。
- **来源溯源：** 每个由 CLI 安装的目录都带一个 `.installed-from` 标记。`list` 用它来显示一个技能从哪来，`uninstall` 用它来区分 CLI 托管的技能和你手写的技能 —— 移除一个**用户手写**的技能需要 `--force`。
- **`git` 是前提。** 安装脚本（`install.sh`）在缺少 git 时会告警并继续，但在它就位之前，技能安装是不会工作的。

:::note[技能在醒来时被快照]
助理在会话开始时捕获每份 `SKILL.md` 正文；会话中途改一个技能，要到**下次**醒来才会改变行为。这是有意为之的 —— 它让被缓存的提示前缀逐字节稳定，于是模型不会在每个回合为一段没变的前缀被重复计费（这也是自动加载记忆槽位背后的同一套道理）。
:::
