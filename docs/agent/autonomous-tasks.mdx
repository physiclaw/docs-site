This is where "I prompt it" becomes "it runs on its own." The built-in
[agent](/en/agent/overview/) doesn't wait for you to ask — **a trigger wakes it, it
operates the phone, then it sleeps.**

## How it wakes

While the server is up, a runtime loop (`agent/runtime/runtime.py`) ticks on a fixed
interval (default **1.0s**), asking each *hook* one question: did anything happen? Nothing
fired? It sleeps and ticks again. When a hook reports a trigger, the loop wakes a fresh
**session** to handle it, then goes back to ticking.

```text
 tick ──► hook: "anything happen?" ──► no ──► sleep ──► tick …
                       │
                       └─ yes ──► wake a session ──► act ──► cooldown ──► tick …
```

Two gates keep the agent asleep until the rig is actually ready:

- **The loop only ticks once `/api/status` reports `ready=true`.** That flag is flipped
  true by the final step of [calibration](/en/set-up/calibrate/). Before that, the server
  can be up and you can still drive it by hand, but the agent stays asleep.
- **A model must be configured.** Even at `ready=true`, the agent will not wake without
  one — set it with `physiclaw models key <provider>` then `physiclaw models use
  <provider/model>` (or `PHYSICLAW_MODEL`). A model-less server still runs for hardware
  setup and manual MCP calls, it just never reacts. See [Models](/en/agent/models/).

## Two kinds of trigger

Each tick polls two hooks. One watches the clock, the other watches the screen:

  <Card title="Cron — on a schedule" icon="seti:clock">
    The cron hook (`agent/hooks/cron.py`) reads `~/.physiclaw/jobs/jobs.md` every tick and
    fires any job whose next fire time has arrived and whose status is `pend`. This is
    recurring work: "check the weather at 8am," "place the grocery order every Friday."
  </Card>
  <Card title="Poll — on a screen change" icon="seti:image">
    The poll hook (`agent/hooks/poll.py`) calls `GET /api/phone/watch` — the overhead
    camera literally watches the phone screen and fires when it changes: a new message, a
    notification, a screen you'd been waiting on. This is reaction, not the clock.
  </Card>
</CardGrid>

## One session at a time

The loop never runs two sessions concurrently. Checking the hooks and running the reaction
are *awaited in sequence*, so no new tick starts mid-reaction. After a reaction finishes,
the loop sleeps `react_cooldown_seconds` (default **6.0s**, `CONFIG.engine
.react_cooldown_seconds`) — long enough for screen animations to settle and for the next
watch poll to re-baseline before it checks again. The agent won't trample its own stylus
moves.

## What a "session" is

A session is one self-contained run of the look → decide → act loop, spun up to handle the
trigger(s) that just fired. Each session starts **fresh** — new conversation, its own
trace — but it isn't amnesiac: at wake it loads the
[persistent memory](/en/agent/memory-and-skills/), and it's told the current date and
exactly which triggers woke it.

Under the hood, the reaction is dispatched one of two ways: for a `claude-code/<model>`
agent it runs Anthropic's `claude` CLI as a subprocess (`spawn_claude`); for any other
provider it runs the in-process engine. Either way the loop above is the same — see
[Inside the engine](/en/agent/inside-the-engine/) for the internals.

A session runs until the agent closes it with a one-word **verdict**:

- **`DONE`** — the task finished.
- **`WAIT`** — work isn't done; resume later (covered below).
- **`FAIL`** — it couldn't complete and isn't going to.
- **`IDLE`** — nothing to do; the wake needed no action (e.g. a spurious phone-watch).
- **`STUCK`** — it ran out of turns, the provider gave out, or it hit an unrecoverable
  blocker. PhysiClaw retries a `STUCK` session a couple of times before accepting it.
  (`DONE` / `IDLE` / `WAIT` / `FAIL` all exit cleanly with no retry.)

## Scheduling a recurring task

Scheduled work lives in one human-readable file, `~/.physiclaw/jobs/jobs.md`. Each
`## <id>` section is a job. **Don't hand-edit fire times** — the easiest way to add a job
is to ask the agent, which uses the bundled `cron` skill to write a correctly-formed entry
and re-verify that the file still parses.

A job looks like this:

```markdown
## morning-weather

Tell me the day's forecast every morning.

- Type: periodic
- Schedule: `0 8 * * *`
- Create time: 2026-06-22T21:40
- Next fire time: 2026-06-23T08:00
- Last fire time: (never)
- Execution time: (never)
- Execution result: (never)
- Status: pend
- Context: Open the weather app, read today's high/low and conditions, and message them to me on WeChat.
```

Three fields carry the meaning:

- **`Schedule`** is standard 5-field cron — `min hour day-of-month month day-of-week` —
  supporting `*`, `N`, `N,M`, `*/N`, and `A-B` (day-of-week `0`=Sun…`6`=Sat). `0 8 * * *`
  is "08:00 every day"; `*/30 * * * *` is "every 30 minutes."
- **`Context`** is the agent's brief for the run — which app to open, what to do, edge
  cases. (The one-line description under the heading is for *you* skimming the file;
  `Context` is for the agent doing the work.)
- **`Type`** is `periodic` (fires every time its schedule matches) or `one-time` (fires
  once, then closes for good). `one-time` is the default for follow-ups, reminders, and
  deferred actions.

:::note[Jobs are never deleted]
A job moves through statuses instead of vanishing: `pend` → `fired` (the hook fired it) →
back to `pend` for a periodic job, or `done` / `fail` for a one-time one. You cancel a job
by setting its status to `cancel`. Terminal jobs (`cancel` / `done` / `fail`) auto-purge
after 7 days. The agent marks completion itself through the cron CLI —
`python -m physiclaw.agent.hooks.cron done|fail|cancel <id> [result summary]` (the runtime
invokes it as `uv run python -m physiclaw.agent.hooks.cron …`).
:::

## "Check back later" — the WAIT verdict

Some tasks can't finish in one pass: you place an order and have to wait for a
confirmation, or you're watching for a reply that hasn't arrived. That's what `WAIT` is
for — the agent ends the session having decided to resume later.

**Normally the agent schedules its own follow-up before closing** — it writes a `one-time`
job for the exact moment it wants to wake. That's the right pattern, because the agent
knows when the next state is likely to be ready.

**Safety net:** if it closes `WAIT` but *forgets* to schedule one, PhysiClaw auto-creates a
single generic follow-up job (canonical id `wait-check-auto`, `agent/engine/jobs.py`) that
fires in **~15 minutes**, re-checks the state, and continues. That generic 15-minute wake
is usually wrong for the specific task — which is exactly why the agent is told to schedule
its own. The same canonical id is reused across WAITs, so the file never grows a new entry
each time a task pauses.

The upshot: a long-running errand survives across sleeps. The agent does what it can now,
parks a wake for later, and the next session picks the thread back up — with memory intact.

## Managing jobs safely

`jobs.md` is plain Markdown, but hand-editing fire times or status fields is the easy way
to corrupt it. Two safe paths:

- **Ask the agent.** The bundled `cron` skill verifies the file parses both before and
  after every change, so a malformed edit can't slip in.
- **Use the cron CLI.** You can inspect the file with the cron skill's `verify` command,
  and the `done|fail|cancel <id>` subcommands manage a job's lifecycle without touching the
  raw fields.

See the [CLI reference](/en/reference/cli/) for the full command surface, and
[Memory & skills](/en/agent/memory-and-skills/) for what the agent carries between these
unattended runs.
